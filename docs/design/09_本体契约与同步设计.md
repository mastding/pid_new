## 0. 文档说明

### 0.1 设计目标

1. 定义系统**依赖的本体实体与字段**，让本体侧可以按契约准备数据；
2. 定义本体**调用规约**（MCP server / tool / 入参 / 出参 / 超时 / 失败行为）；
3. 给出**字段级读写矩阵**：哪个阶段在哪个节点读哪些字段、写哪些字段；
4. V3.0 仅做"读 + 本地暂存"，V3.1 接通"写回 MCP"，本文档同时覆盖两阶段；
5. 给出**降级矩阵**：本体不可达 / 字段缺失 / 内容矛盾时系统怎么继续。

### 0.2 设计原则

- **字段级粒度**：契约描述到字段，不是到实体；同一实体的字段可独立缺失/降级；
- **可追溯**：每个使用本体值的决策点必须把 `evidence` 字段填上 `source="ontology", path="loop.pv.usl"`；
- **不可信输入**：本体内容必须经结构化字段化才能进入 LLM prompt 或决策；不允许整段 plain-text 注入 prompt；
- **本地兜底优先于猜测**：缺字段时优先用本地配置默认 / `policies/loop_priors.py` 兜底，而不是用 LLM"估"；
- **写回是显式动作**：任何写回都需要工程师确认 + 审计，不允许系统隐式写回。

### 0.3 名词

| 名词 | 含义 |
|---|---|
| 本体（Ontology） | 工艺/设备/控制层面的结构化知识，包括回路定义、变量定义、阈值、耦合关系等 |
| MCP | Model Context Protocol；本系统统一通过 MCP 客户端访问本体 |
| MCP server | 提供本体能力的服务实例，可有多个（如 ontology_main、ontology_legacy） |
| MCP tool | 一个 server 暴露的具体能力，如 `get_loop`、`list_conditions` |
| 字段路径 | 类似 `loop.pv.usl` 的点分路径，唯一标识一个字段 |
| 兜底（fallback） | 本体缺失时使用的默认值或推断值 |
| 写回（writeback） | 把系统计算的结果（如 K/T/L）回填到本体 |

## 1. 本体实体定义

### 1.1 实体概览

V3.0 用到的本体实体：

```
Loop ──┬── Variable (PV / MV / SV)
       ├── Condition[]            (工况列表)
       ├── KpiThreshold           (Cpk / Harris 阈值，可选按 condition)
       ├── Coupling[]             (耦合关系，V3.1 用)
       ├── DvVariable[]           (扰动变量，V3.1 用)
       └── ProcessModel           (历史 K/T/L，V3.0 只读，V3.1 写回)
```

### 1.2 Loop（回路）

```yaml
Loop:
  loop_id: str                # 例 "5203_TIC_10707"
  loop_name: str              # 例 "3床冷氢"
  loop_type: enum             # flow | pressure | temperature | level
  description: str
  plant_id: str               # 装置/车间标识
  controller_type: enum       # PI | PID | PI_D | ...
  control_direction: enum     # direct | reverse
  pv: VariableRef
  mv: VariableRef
  sv: VariableRef | null
  dv_list: list[VariableRef] = []   # V3.1
  conditions: list[ConditionRef] = []
  coupling: list[CouplingRef] = []
  kpi_thresholds: KpiThresholdSet | null
  process_model_history: list[ProcessModelSnapshot] = []
  meta:
    last_updated_at: datetime
    last_updated_by: str
    sources: list[str]         # ["岗位操作法 v3.2", "DCS 组态 2026-Q1"]
```

### 1.3 Variable（变量）

```yaml
Variable:
  tag: str                    # DCS 位号
  name: str
  unit: str                   # 工程单位
  range:
    min: float
    max: float
  usl: float | null           # 上规格界限（仅 PV 必需）
  lsl: float | null           # 下规格界限
  alarm_high: float | null    # 报警上限
  alarm_low: float | null
  raw_data_source:            # 历史数据接入信息
    system: str               # "Hollysys" | "Honeywell" | ...
    path: str
  filtering:                  # 当前 DCS 滤波配置（参考用）
    type: enum                # none | first_order | sma
    tau_s: float | null
```

### 1.4 Condition（工况）

```yaml
Condition:
  id: str                     # 例 "low_load"
  name: str                   # 例 "低负荷"
  description: str
  match_features:             # Node-1 工况识别用
    - key: str                # 如 "pv_stats.mean"
      range: [float, float]
      weight: float = 1.0
  applicable_thresholds:      # 该工况下的阈值覆盖
    cpk_threshold: float | null
    harris_threshold: float | null
  applicable_constraints:     # 该工况下的工艺约束
    max_step_amplitude_pct: float | null   # 最大可接受 SP 阶跃幅度（V3.1 阶跃测试用）
    expected_settling_time_s: float | null
```

### 1.5 KpiThresholdSet（阈值集合）

```yaml
KpiThresholdSet:
  default:
    cpk_threshold: float = 1.0
    cpk_excellent: float = 1.33
    harris_threshold: float = 0.6
    harris_excellent: float = 0.8
  per_condition:              # 可选，覆盖 default
    - condition_id: str
      cpk_threshold: float | null
      harris_threshold: float | null
```

### 1.6 Coupling（耦合关系，V3.1）

```yaml
Coupling:
  id: str
  source_loop_id: str
  target_loop_id: str
  type: enum                  # disturbance | resonance | cascade
  strength_hint: enum         # weak | medium | strong | unknown
  description: str
  source_evidence: list[str]  # 引用来源
```

### 1.7 ProcessModelSnapshot（历史模型，V3.0 只读）

```yaml
ProcessModelSnapshot:
  loop_id: str
  condition_id: str | null
  model_type: enum            # FOPDT | SOPDT | ...
  K: float
  T: float
  L: float
  T1: float | null
  T2: float | null
  zeta: float | null
  identified_at: datetime
  identified_by: str          # "system_v2" | "engineer_jane" | ...
  source: str                 # 哪次 session
  confidence: float
```

## 2. MCP 调用规约

### 2.1 Server 配置

延续现有 `mcp_config`：

```yaml
servers:
  - name: ontology              # 主本体服务
    transport: stdio | ws | http
    endpoint: "..."
    enabled: true
    timeout_s: 5.0
    auth: { type: token, token: "..." }   # 通过 var/config
  - name: ontology_legacy       # 备用旧本体（可选）
    enabled: false
    ...
```

### 2.2 Tool 命名约定

每个 tool 名形如 `<动作>_<实体>`，全小写下划线：

| Tool 名 | 实体 | 动作 | V3.0 必需 |
|---|---|---|---|
| `get_loop` | Loop | 读 | ✅ |
| `list_conditions` | Condition[] | 读 | ✅ |
| `get_pv_limits` | Variable.usl/lsl | 读 | ✅ |
| `get_mv_range` | Variable.range | 读 | ✅ |
| `get_kpi_thresholds` | KpiThresholdSet | 读 | ✅ |
| `list_couplings` | Coupling[] | 读 | V3.1 |
| `list_dv_variables` | DvVariable[] | 读 | V3.1 |
| `get_process_model_history` | ProcessModelSnapshot[] | 读 | ✅（仅读） |
| `update_process_model` | ProcessModelSnapshot | 写 | V3.1 |
| `update_pv_limits` | Variable.usl/lsl | 写 | V3.1 |

### 2.3 Tool 入参 / 出参 schema

> 每个 tool 都用 OpenAI function-calling 兼容的 JSON Schema 描述。完整 schema 见附录 A。

示例 `get_loop`：

```yaml
input:
  type: object
  properties:
    loop_id: { type: string }
  required: [loop_id]

output:
  type: object
  properties:
    found: { type: boolean }
    loop:  { type: object, $ref: "#/Loop" }
    error: { type: string, nullable: true }
  required: [found]
```

### 2.4 调用约定

| 项 | 默认 | 备注 |
|---|---|---|
| 单次超时 | 5 s | 配置可调 |
| 重试 | 1 次（指数退避 1s） | 写操作不重试 |
| 并发 | 同一会话内串行 | 跨会话不限 |
| 缓存 | 本地内存 LRU，TTL = 600 s | 见 § 5 |
| 失败传播 | 不抛异常，返回 `{found:false, error}` | 由调用方决策 |

### 2.5 错误码

| 错误码 | 含义 | 系统侧动作 |
|---|---|---|
| `not_found` | 实体不存在 | 走 fallback |
| `field_missing` | 实体存在但字段缺失 | 走 fallback + risk_flag |
| `invalid_data` | 数据矛盾（USL ≤ LSL 等） | AlarmEvent("invalid_ontology_data") |
| `timeout` | 超时 | 重试 1 次后走 fallback |
| `auth_failed` | 鉴权失败 | error_event，整体 abort |
| `server_unreachable` | server 不可达 | fallback + risk_flag="ontology_unreachable" |

### 3.2 写矩阵（V3.0：仅写本地；V3.1：写回 MCP）

| 字段路径 | 谁写 | V3.0 落点 | V3.1 写回 MCP |
|---|---|---|---|
| `loop.process_model_history` 追加新 snapshot | 整定流水线成功结束 | `var/ontology_writeback/<loop_id>/<ts>.json` | `update_process_model` |
| `loop.pv.usl` / `lsl` | 工程师在 confirmation 中填写并勾选 writeback | 同上 | `update_pv_limits` |
| `loop.mv.filtering`（推荐档位） | 诊断 SNR 低 + 工程师确认采用 | 同上 | （V3.x） |

### 3.3 读取入口集中

所有读取通过 `core/ontology/loader.py` 的统一接口：

```python
class OntologyLoader:
    async def load_loop(self, loop_id: str) -> LoopOntology: ...
    async def load_pv_limits(self, loop_id: str) -> PvLimits | None: ...
    async def load_kpi_thresholds(self, loop_id: str, condition_id: str | None) -> KpiThresholdSet: ...
    async def load_conditions(self, loop_id: str) -> list[Condition]: ...
    ...
```

`OntologyLoader` 负责：

- 调度 MCP tool；
- 缓存；
- 失败降级；
- 给每条返回值附上 `source=ontology, path=...` 元数据，供 evidence 引用。

## 4. 调用时序

### 4.1 诊断流水线一次完整调用（happy path）

```
DiagnosticPipeline.start(loop_id="TIC-10707")
  │
  ├── OntologyLoader.load_loop(loop_id)
  │     → MCP get_loop
  │     → 命中缓存？是 → 返回；否 → 调用 → 写缓存
  │
  ├── Node-1 identify_condition
  │     ├── load_conditions(loop_id) ← 已在 load_loop 时一并取，从缓存读
  │     └── 计算工况
  │
  ├── Node-2 detect_open_loop
  │     ├── load_mv_range(loop_id) ← 缓存
  │     └── 算法
  │
  ├── Node-3 estimate_snr
  │     └── 不依赖本体（仅按 loop_type 取默认）
  │
  ├── Node-4 compute_cpk
  │     ├── load_pv_limits(loop_id) ← 缓存命中？返回；缺字段 → 触发 confirmation
  │     ├── load_kpi_thresholds(loop_id, condition_id) ← 缓存
  │     └── 算法
  │
  ├── Node-5 compute_harris_closed_loop
  │     ├── load_kpi_thresholds(...) ← 缓存
  │     └── 算法
  │
  └── Node-6 summarize → verdict
```

整个诊断会话内本体调用次数 = MCP tool 数（约 4-6 次），后续节点全部命中缓存。

### 4.2 缺字段触发 confirmation

```
Node-4 compute_cpk
  ├── load_pv_limits(loop_id) → {usl: null, lsl: 50}
  ├── 缺 usl → produce EngineerConfirmation("missing_pv_limits", evidence={ontology_path: "loop.pv.usl"})
  ├── persist confirmation → var/confirmations/...
  ├── persist session snapshot
  └── SSE engineer_confirmation_required → 关闭流

工程师 →
  POST /api/confirmations/{id}/respond { usl: 200, lsl: 50, writeback_to_ontology: true }

System →
  - confirmation.status = responded
  - 把 response 写入 var/ontology_writeback/<loop_id>/pv_limits_<ts>.json
    （V3.0 不调 MCP；V3.1 调 update_pv_limits）

  POST /api/sessions/{id}/resume
    → 重入 Node-4，把 response.usl/lsl 注入入参
    → 计算 Cpk
    → confirmation.status = applied
```

## 5. 缓存策略

### 5.1 缓存对象

| Key | 内容 | TTL |
|---|---|---|
| `loop:<loop_id>` | 整个 Loop 实体 | 600 s |
| `loop:<loop_id>:pv_limits` | PV USL/LSL | 600 s |
| `loop:<loop_id>:kpi:<condition_id>` | 阈值 | 600 s |
| `loop:<loop_id>:conditions` | 工况列表 | 600 s |
| `loop:<loop_id>:model_history` | 历史模型 | 60 s（更新频繁） |

### 5.2 失效

- TTL 到期；
- 同一进程内显式 `OntologyLoader.invalidate(loop_id)`；
- 工程师 confirmation 触发 writeback 时，立即 invalidate 该 loop 的相关 key。

### 5.3 一致性

- 单进程：内存 LRU 即可；
- 多进程（生产）：用 SQLite WAL 在 `var/ontology_cache.db`，跨进程一致；
- 不做分布式缓存（V3.0 单实例部署）。

## 6. 失败降级矩阵

### 6.1 总体规则

```
本体调用结果分三类：
  A. 成功 + 字段齐全          → 正常使用
  B. 成功 + 字段缺失           → fallback + risk_flag
  C. 失败（超时/不可达/auth） → fallback + risk_flag="ontology_unreachable"
                                + 累计 N 次后 → AlarmEvent
```

### 6.2 字段缺失 fallback 表

| 缺失字段 | 兜底来源 | risk_flag |
|---|---|---|
| `loop.loop_type` | 用前端入参 / 数据画像推断 | `loop_type_inferred` |
| `loop.controller_type` | 默认 "PI" | `controller_type_default` |
| `loop.control_direction` | 默认 "direct" | `control_direction_default` |
| `loop.pv.range` | 用数据 max-min | `pv_range_inferred` |
| `loop.pv.usl/lsl` | **不能 fallback**，必须 confirmation | — |
| `loop.mv.range` | 用数据 max-min | `mv_range_inferred` |
| `loop.conditions` | 单工况 "default" | `no_condition_in_ontology` |
| `loop.kpi_thresholds.default` | 用配置默认（Cpk=1.0, Harris=0.6） | `kpi_threshold_default` |
| `loop.process_model_history` | 空列表 | （无） |

### 6.3 重复失败的告警阈值

```
若 10 分钟内同一 server 失败 ≥ 5 次：
  → AlarmEvent("ontology_server_unstable", severity=high)
  → 记入运维监控

若 24 小时内某字段 fallback 次数 ≥ 50:
  → 提示运维"该字段在本体中可能批量缺失，建议补全"
```

### 6.4 矛盾数据处理

| 矛盾 | 行为 |
|---|---|
| `usl ≤ lsl` | AlarmEvent("invalid_ontology_data"), 节点中止，整体 abort |
| `range.min ≥ range.max` | 同上 |
| `mean(PV) 超出 [lsl, usl]` 但仍在 range 内 | risk_flag="pv_out_of_spec_in_data"，不阻断（这本身就是 Cpk 低的体现） |
| `condition.match_features.range` 互相冲突（多个工况覆盖同一 PV 区间） | 取 weight 最大的，risk_flag="condition_overlap" |

## 7. 写回设计

### 7.1 V3.0：本地暂存

写回目录结构：

```
var/ontology_writeback/
├── <loop_id>/
│   ├── pv_limits_<ts>.json          # 工程师补充 USL/LSL
│   ├── condition_<ts>.json          # 工程师补充 condition 定义
│   ├── process_model_<ts>.json      # 整定结果沉淀
│   └── ...
└── pending_index.jsonl              # 待提交清单
```

每个文件 schema：

```json
{
  "loop_id": "TIC-10707",
  "type": "pv_limits | condition | process_model | mv_range | filtering",
  "payload": { ... },                # 与本体实体字段一致
  "evidence": {
    "session_id": "...",
    "trace_id": "...",
    "confirmation_id": "..." | null,
    "tuning_session_id": "..." | null
  },
  "user": "engineer_jane",
  "timestamp": "2026-05-08T13:25:00Z",
  "submitted_to_mcp": false,         # V3.0 全部 false
  "submitted_at": null
}
```

### 7.2 V3.1：MCP 写回

V3.0 不实现，但需为它预留：

- `OntologyWriter.flush_pending(loop_id)`：批量提交；
- 写回前再次校验本体侧字段未在期间被他人修改（乐观锁，对比 last_updated_at）；
- 失败时保留本地暂存文件，标 `last_attempt_at` 与 `last_attempt_error`；
- 提供 CLI：`python -m core.ontology.cli flush --loop_id ...`。

### 7.3 审计

每条写回（即使是本地）必须落审计：

```
var/audit/ontology_writeback.jsonl
{ts, user, loop_id, type, action: "stage" | "submit" | "rollback", session_id, trace_id}
```

### 7.4 工程师确认与写回的关系

| 场景 | 写回？ | 时机 |
|---|---|---|
| 整定成功 | 是（K/T/L） | 整定 result_event 之后，**异步**沉淀到本地，不阻塞流水线 |
| 工程师补 USL/LSL 并勾选 writeback | 是 | confirmation.responded 之后立即沉淀 |
| 工程师补充但未勾选 writeback | 否 | 仅在本会话生效，会话结束即丢弃 |
| LLM Advisor 推断的值 | **永不写回** | 仅作为运行时提示 |

## 8. 安全

### 8.1 防注入

- 本体返回的字符串字段（description / name / hint）**不直接拼到 LLM system prompt**；
- 必须先做：
  1. 长度截断（单字段 ≤ 500 字符，整体 ≤ 4000 字符）；
  2. 移除控制字符与 prompt-injection 模式（`ignore previous` 等关键字标记并替换为占位符）；
  3. 包装在 JSON 字段里传给 LLM，让 LLM 把它当数据而非指令。

### 8.2 鉴权

- MCP server 通过 token 认证，token 在 `var/config/mcp_credentials.json`；
- Token 不进日志、不进会话快照、不返回前端；
- V3.0 单租户，无 RBAC；V3.x 接 SSO 后做按角色读写授权。

### 8.3 写回授权

V3.0 本地写回不需要额外授权（系统单租户）；V3.1 写回 MCP 需：

- 工程师确认事件 `response_user` 字段必填且非空；
- MCP server 侧二次校验 user 写权限；
- 写回失败的告警通知到运维。

### 8.4 敏感字段

V3.0 不涉及敏感个人信息。本体中如出现联系方式 / 操作员姓名等，按字段标记 `pii=true`，从前端展示中脱敏。

## 附录 C：本体侧对接 FAQ

**Q1：必须用 MCP 吗？**
A：是。统一通过 MCP 让本系统可以无代码切换不同本体源。如对接旧系统，可写一个 MCP adapter。

**Q2：是否可以一次返回所有字段？**
A：`get_loop` 已聚合大部分字段。其余 tool 是为了让 V3.0 之外的轻量调用避免拉太多。

**Q3：写回是否可以异步？**
A：V3.1 的写回采用"工程师点确认 + 系统异步提交"，但提交结果必须能回查（`pending_index.jsonl`）。

**Q4：本体能否反向通知系统字段更新？**
A：V3.0 不支持。V3.x 评估 webhook。

**Q5：缓存 TTL 600s 会不会太久？**
A：可调。整定/诊断阶段单次会话内通常 <60s，TTL 取 600s 是为了多会话短期复用；如本体更新频繁，可调到 60s。
