## 0. 文档说明

### 0.1 设计目标

1. 锁定 **V3.0 全部数据契约**：实体、字段、类型、约束、可空性；
2. 给出**实体关系图（ER）**，让前后端能共享一份心智模型；
3. 给出**状态机**：Session / AlarmEvent / EngineerConfirmation / TuningRound / DiagnosticVerdict；
4. 给出**持久化方案**：`var/` 目录布局、文件格式、索引结构、生命周期；
5. 给出**数据隔离与一致性**约定，避免跨模块踩坑。

### 0.2 设计原则

- **强类型 Pydantic**：跨层调用一律走模型，不传裸 dict；
- **字段稳定优先**：新增字段允许，删除字段需 ADR；
- **可空字段必须显式标注**：`Optional[X] = None` 而不是默认值；
- **时间一律用 UTC ISO8601**：前端转地区时区显示；
- **id 一律 uuid v4 字符串**：除 `loop_id` 是业务字段外；
- **审计无脱漏**：任何状态机迁移都在 jsonl 留痕。

### 0.3 名词

详见 PRD § 0.3 与文档 9 § 0.3。

## 3. Pydantic 数据契约

> 所有模型在 `backend/models/`。
> 命名约定：实体本身用 `Entity`；请求体用 `<Action>Request`；响应用 `<Action>Response`。

### 3.1 通用基础

```python
# models/_base.py

from datetime import datetime
from typing import Any, Literal
from pydantic import BaseModel, Field
from uuid import uuid4

class TimestampedModel(BaseModel):
    created_at: datetime
    updated_at: datetime

def gen_id() -> str:
    return str(uuid4())

class Evidence(BaseModel):
    """统一证据结构，所有诊断/告警/确认都用它引用数据来源。"""
    fact: str                            # 简述
    value: Any | None = None             # 数值
    source: Literal["data", "ontology", "config", "policy", "llm"]
    path: str | None = None              # 例 "loop.pv.usl" 或 "data_profile.pv_stats.mean"
    extra: dict[str, Any] = {}
```

|---|---|---|
| 单实体 (`*.json`) | UTF-8 JSON，缩进 2 | 写时原子 rename |
| 索引 (`index.jsonl`) | JSON Lines，append-only | 任何变更追加一条 |
| 事件流 (`events.jsonl`) | 同上 | 一行一事件 |
| 缓存 (`*.db`) | SQLite WAL | 多进程安全 |

### 5.3 索引 schema

```jsonc
// var/sessions/index.jsonl
{"ts":"2026-05-08T13:00:00Z","action":"created","session_id":"...","pipeline":"diagnostic","loop_id":"..."}
{"ts":"2026-05-08T13:01:30Z","action":"completed","session_id":"...","status":"completed","duration_ms":2300}
{"ts":"2026-05-08T13:02:10Z","action":"awaiting_confirmation","session_id":"...","confirmation_id":"..."}
```

```jsonc
// var/alarms/index.jsonl
{"ts":"...","action":"created","alarm_id":"...","loop_id":"...","type":"open_loop","severity":"high"}
{"ts":"...","action":"acknowledged","alarm_id":"...","ack_user":"li"}
{"ts":"...","action":"resolved","alarm_id":"...","resolution_note":"..."}
```

### 5.4 写入原子性

- 单文件写入：先写 `<file>.tmp`，再 `os.replace` 到目标；
- 索引追加：用 `O_APPEND | O_CREAT`，单写者保证顺序；
- SSE 事件落 jsonl 与发送给客户端**同一事务**：先写文件再发 SSE，避免重连时缺事件。

### 5.5 大数据字段处理

| 字段 | 存储策略 |
|---|---|
| `cleaned_df` 时序 | **不持久化**，每次重入重新加载 csv |
| `time_series_preview`（评估用，t/sp/pv/mv 各 ≤ 500 点） | 进 `tuning_results/*.json` |
| `attempts`（含 `fit_preview`） | 仅保留 top-N（默认 N=20） |
| LLM `reasoning_content` | 单独存 `llm_traces/<trace_id>.json` |
| MCP 原始返回 | 不持久化（仅落 ontology_meta 摘要） |

### 5.6 数据生命周期

| 数据 | TTL | 清理策略 |
|---|---|---|
| `sessions/<id>/events.jsonl` | 90 天 | 后台 cron 压缩为 tar.gz 移到 `archive/` |
| `sessions/<id>/llm_traces/*` | 30 天 | 同上 |
| `alarms/`（resolved/dismissed） | 365 天 | 归档 |
| `alarms/`（new/acknowledged） | 永久 | 不清理 |
| `experience/` | 永久 | 不清理 |
| `audit/` | 永久 | 不清理 |
| `logs/` | 30 天 | 按文件大小滚 |
| `ontology_cache.db` | TTL 内 | 进程内自动失效 |

清理任务：`backend/scripts/cleanup_var.py`，由部署 cron 调度（V3.0 手动；V3.x 进程内任务）。

## 6. 数据隔离与一致性

### 6.1 隔离

| 维度 | 是否隔离 |
|---|---|
| 不同 session | 完全隔离（独立目录） |
| 同 loop 不同 session | 独立目录，但 alarm 可关联（related_alarm_ids） |
| 不同 loop | 完全隔离 |
| 配置 | 单全局；写时审计 |

### 6.2 一致性

| 场景 | 措施 |
|---|---|
| Session ↔ Confirmation | 状态机驱动；`SessionState.pending_confirmation_id` 与 `EngineerConfirmation.status=pending` 必须一致；后台任务检测漂移 |
| Session ↔ Alarm | 由 session 创建告警；告警的 `session_id` 必须存在 |
| AlarmEvent.status 与 audit.jsonl | 每次状态变更同时写两个文件，写失败时回滚（先 audit 后 主文件，主文件失败时审计仍保留为"未完成迁移"标记） |
| Experience ↔ TuningResult | Experience 在整定 result_event 之后异步写；不阻塞主流程 |
| OntologyWriteback.payload ↔ 本体 schema | V3.0 仅本地校验；V3.1 写回前 round-trip 校验 |

### 6.3 重入一致性

```
触发挂起时：
  1. 写 EngineerConfirmation (status=pending)
  2. 写 SessionState (status=awaiting_confirmation, current_node, completed_nodes, node_outputs, input_hash)
  3. 推 SSE engineer_confirmation_required
  4. 关闭 SSE 流

触发 resume 时：
  1. 加载 SessionState
  2. 加载 EngineerConfirmation (校验 status in {responded})
  3. 校验 csv 的 input_hash 仍一致（变了 → 拒绝 resume，confirmation 标 expired）
  4. 重建 LoopContext，注入 confirmation.response
  5. 节点幂等重跑
  6. confirmation.status = applied，session.status = running
  7. 后续节点继续
```

### 6.4 引用完整性

- `Session.tuning_result_id` 指向的文件必须存在；
- `Session.diagnostic_verdict_id` 同上；
- `Experience.tuning_session_id` 必须能找到对应 session；
- 删除文件前必须更新引用（V3.0 不删）。

## 7. 数据契约演进

### 7.1 兼容性原则

| 变更类型 | 需要 |
|---|---|
| 新增可选字段 | 不需要任何额外动作 |
| 新增必填字段 | 需要 ADR + 数据迁移脚本 |
| 删除字段 | 需要 ADR + 至少 1 个版本兼容期 |
| 重命名字段 | 等价于"加新 + 弃旧"，需 ADR |
| 类型变更 | 需 ADR + 迁移 |

### 7.2 版本字段

```python
class Envelope(BaseModel):
    schema_version: str = "v3.0"
    payload: Any
```

V3.0 内部不需要 envelope；跨版本读旧文件时，由 `models/_migrations/` 提供 reader：

```python
def read_alarm(path: str) -> AlarmEvent:
    raw = json.loads(...)
    version = raw.get("schema_version", "v3.0")
    return MIGRATORS[version](raw)
```
