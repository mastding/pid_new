## 0. 文档说明

### 0.1 设计目标

1. 把"一次完整的 PID 整定"拆解为可独立测试、可重放、可替换的阶段；
2. 算法与编排彻底解耦：算法在 `algorithms/` 内是纯函数，编排在 `pipeline/runner.py` 内只描述串行顺序；
3. 通过 `Skill / Provider / Policy` 三层抽象，让任意阶段（窗口检测、辨识、整定、评估）的实现可以在不动流水线的情况下被替换；
4. LLM 决策点（窗口策略生成、窗口选择、辨识评审、辨识精修）以"可拔插顾问"形式接入流水线，失败必须回退到确定性路径；
5. 所有阶段产出 SSE 事件供前端渲染，结构对前端契约稳定。

### 0.2 设计原则

- **确定性优先**：每一阶段都先有"无 LLM 也能跑出合理结果"的确定性实现，LLM 仅做增量改善；
- **纯函数化**：`core/algorithms/` 不持有状态、不调外部服务，便于单元测试；
- **Skill 是 LLM 视角的能力，Provider 是算法视角的实现**：Skill 的入参由 LLM 提供并经 Pydantic 校验；Provider 是同一类能力下的算法实现，由 Skill 选择并调用；
- **Policy 是规则与先验**：把"loop_type 优先模型"、"窗口算法族开关"、"质量分加权"这类经验抽到 `policies/`，与算法解耦；
- **失败不中断**：每个 LLM 调用都有确定性回退；每个 Skill 失败都返回 `SkillResult(success=False)` 而非抛异常。

### 0.3 名词约定

| 名词 | 含义 |
|---|---|
| **Loop（回路）** | 一组 PV/MV/SV 时序数据，对应一个被控对象 |
| **Loop type** | flow / pressure / temperature / level，决定先验与调参偏好 |
| **Window（窗口）** | 时序中一段适合做辨识的连续区间 |
| **算法族（family）** | 窗口检测的策略家族：`sp_step / mv_step / mv_ramp / steady_disturbance / rolling_scan` |
| **Process model** | 过程模型：FO / FOPDT / SOPDT / SOPDT_UNDER / IPDT / IFOPDT |
| **Provider** | 一种能力的具体算法实现（如 `transfer_function_fit`） |
| **Skill** | LLM 可调用的原子能力（如 `identify_model`），内部装配 Provider |
| **Policy** | 规则/先验模块（如 loop_priors、constraints、scoring_rules、refinement） |
| **Stage event** | 流水线阶段事件，类型为 `running / done / error` |

## 1. 流水线总览

### 1.1 端到端流程图

```
┌──────────────────────────────────────────────────────────────────────────────┐
│ run_tuning_pipeline (AsyncGenerator[stage_event | result_event | error_event])│
└──────────────────────────────────────────────────────────────────────────────┘
        │
        ▼
  ┌───────────────┐
  │ 1. data_analysis │ load_dataset（Skill）→ 清洗 + dt
  └───────┬───────┘
          │ + 数据画像（loop_features / summarize_data）
          ▼
  ┌───────────────────┐
  │ 1.b ontology_policy│ 本体/MCP 上下文 → build_window_selection_policy
  │                    │ ↳ 可选 LLM 改写策略（ask_window_policy_via_llm）
  └────────┬───────────┘
           │ window_policy（merge_gap_s / disabled_families / hints）
           ▼
  ┌──────────────────┐
  │ 1.c detect_windows │ Skill → policy_composite Provider
  └────────┬─────────┘
           │ candidate_windows + step_events + family_summaries
           ▼
  ┌──────────────────┐
  │ 1.5 window_selection│
  │   ├── apply_window_policy_to_candidates（重打分）
  │   ├── deterministic：quality_score_selector
  │   └── 可选 LLM：choose_window_via_llm（reasoner 思考链）
  └────────┬─────────┘
           │ chosen_window + window_candidate_decision
           │（formal_identification_allowed 决定是否能进 Stage 2）
           ▼  early-stop: stop_after == "window_selection"
  ┌────────────────────────────────────────┐
  │ 2. identification（最多 1 + 2 次精修）│
  │   for round in 0..MAX_REFINEMENT_ROUNDS:│
  │     ├── identify_model Skill → transfer_function_fit Provider
  │     ├── stage_event identification:done（attempts + algorithm_comparison）
  │     ├── 可选 LLM 评审：review_identification_via_llm
  │     │     verdict in {accept, downgrade}
  │     └── if downgrade: ask_refinement_via_llm
  │             ↳ 失败时退回 recommend_refinement_from_algorithm_comparison
  │   end                                                                  │
  │   收敛策略：accept 立即收敛；耗尽轮次走"跨轮挑最高 fit_score"          │
  └────────┬───────────────────────────────┘
           ▼ early-stop: stop_after == "identification"
  ┌──────────────────┐
  │ 3. tuning         │ generate_tuning_candidates Skill → classic_family Provider
  └────────┬─────────┘
           │（IMC / Lambda / ZN / CHR 候选 + 启发式推荐）
           ▼
  ┌──────────────────┐
  │ 4. evaluation     │ evaluate_tuning Skill → closed_loop_sim + reality_check
  └────────┬─────────┘
           ▼
  result_event（一次性发送全部聚合结果）
```

### 1.2 阶段事件契约

| stage 名称 | running data | done data 关键字段 |
|---|---|---|
| `data_analysis` | — | data_points / sampling_time / candidate_windows / window_detection_meta |
| `ontology_policy` | — | policy / source（default \| llm）/ confidence / ontology_source |
| `window_selection` | — | mode / chosen_index / deterministic_index / window_candidate_decision / window_judgements |
| `identification` | round / max_rounds / refinement | attempts / algorithm_comparison / model_type / K / T / L / r2_score / confidence |
| `model_review` | round | verdict（accept\|downgrade）/ reason / concerns / fallback |
| `identification_refinement` | round | retry / source / force_window_index / force_model_types / hint_L |
| `tuning` | — | strategy / Kp / Ki / Kd |
| `evaluation` | — | passed / performance_score / final_rating / overshoot_percent |

### 1.3 早停（stop_after）

| 取值 | 用途 | 流水线结束位置 |
|---|---|---|
| `None` | 完整跑完整定 | 4. evaluation 完成 |
| `"window_selection"` | "数据分析"菜单 | 1.5 完成后立即发 result |
| `"identification"` | "系统辨识"菜单 | Stage 2 收敛后立即发 result |

### 2.1 数据加载与画像

#### 2.1.0 输入 / 输出

- **输入**：`csv_path`、可选 `loop_prefix`、`loop_type`
- **输出**：
  - `cleaned_df`（pandas DataFrame，列含 PV / MV / SV / time）
  - `dt`：采样周期（s）
  - `data_profile`（pv_stats / mv_stats / data_quality / data_profile / pv_mv_relation_raw / text_summary）
- **入口**：
  - Skill：`load_dataset`（在 `data_understanding/load_dataset.py`）
  - 退化：`load_and_prepare_dataset`（`algorithms/data_analysis.py`）

#### 2.1.1 默认实现

1. `load_dataset` 通过 `dataset_loader` 共享 Provider 读取 CSV，按列前缀解析多回路；
2. 清洗阶段：剔除 NaN、重复时间戳、按时间排序、推断 `dt`（取相邻间隔的中位数）；
3. 画像阶段：调用 `extract_loop_features`，输出 `data_profile`（含 PV/MV span、饱和占比、过程方向、置信度、文字摘要）。

#### 2.1.2 关键约束

- 行数 < 50 → 抛 `DATA_ERROR`；
- 采样周期推断在 `dt` 抖动 > 5% 时记入 `data_quality.warnings`；
- `text_summary` 为 LLM 只读视图，长度 <= 200 字。

#### 2.1.5 异常路径

- `load_dataset` Skill 失败 → 回退 `load_and_prepare_dataset`（纯函数）；
- 仍失败 → `error_event(stage="data_analysis", error_code="DATA_ERROR")`。

### 2.2 本体策略生成（ontology_policy）

#### 2.2.0 输入 / 输出

- **输入**：`loop_name`、`loop_type`、`data_profile`、可选 `frontend_context`、可选 MCP 上下文
- **输出**：`window_policy`（dict），关键字段：
  - `algorithm_plan: list[{family, state, reason}]`
  - `disabled_algorithm_families: list[str]`
  - `merge_gap_s: float`
  - `expected_gain_sign: -1 | 0 | 1 | "unknown"`
  - `expected_time_constant_range_s: [low, high]`
  - `expected_dead_time_range_s: [low, high]`
  - `confidence: float`
  - `ontology_facts: { source, items }`

#### 2.2.1 默认实现

`build_window_selection_policy`（`pipeline/ontology_policy_builder.py`）：

1. 默认开启全部 5 个算法族；
2. 根据 `loop_type` 注入时间常数/死区先验（取自 `policies/loop_priors.py`）；
3. 根据 PV/MV 饱和比、过程方向估计 `expected_gain_sign`；
4. 若有 MCP 本体内容，追加 `ontology_facts`，并允许其覆盖 `disabled_algorithm_families`、`expected_*_range_s`。

#### 2.2.4 LLM 顾问接入

- 仅当 `use_llm_advisor=True`；
- `ask_window_policy_via_llm` 输入 `(base_policy, data_profile, mcp_context, frontend_context)`，输出经 schema 校验的新 `policy`；
- 必须**完全替换**旧策略字段，但保留 `confidence` 与 `ontology_facts.source`；
- LLM 思考链以 `llm_thinking{stage="ontology_policy"}` 单独事件发出，不污染 stage_event。

#### 2.2.5 异常路径

| 失败 | 回退 |
|---|---|
| LLM 不可达 / 超时 | 直接使用 `base_policy`，policy_source="default" |
| LLM 返回 schema 不合法 | 同上，policy_source="default" |
| MCP 本体调用失败 | `ontology_meta.ontology_mcp_error` 记录，仍可生成 default policy |

### 2.3 窗口检测（detect_windows）

#### 2.3.0 输入 / 输出

- **输入**：`cleaned_df`、`dt`、`loop_type`、`window_policy`
- **输出**：
  - `candidate_windows: list[dict]`（每条至少含 `window_start_idx / window_end_idx / window_source / window_algorithm / window_quality_score / window_usable_for_id / window_score_breakdown`）
  - `step_events`（用于 SP 阶跃可视化）
  - `meta.family_summaries`

#### 2.3.1 默认实现 — Provider 总览

| Provider 名称 | 职责 | 作用域 |
|---|---|---|
| `policy_composite` | 默认聚合器：按 `algorithm_plan` 分发到下层 family，再合并 | window_detection（默认） |
| `event_window_builder` | 由 SP/MV 阶跃事件构造窗口 | sp_step / mv_step |
| `history_rule_based` | 按 `window_policy` 应用算法计划与禁用规则 | 所有 family |

`policy_composite.detect()` 流程：

1. 解析 `policy.algorithm_plan` → `(runnable, disabled)`；
2. 按家族顺序 `sp_step → mv_step → mv_ramp → steady_disturbance → rolling_scan` 调用各算法族实现，跳过 disabled；
3. 用 `_merge_events` 按 `merge_gap_s` 合并相邻窗口；
4. 以 `event_window_builder` 把事件展开成完整窗口；
5. 应用 `_apply_algorithm_plan` 给每个窗口注入 `window_algorithm_plan_state` / `window_algorithm_plan_reason`；
6. 计算 `window_quality_score`（见 2.3.3）。

#### 2.3.2 关键算法（节选）

**SP 阶跃检测**（sp_step）：

```
for i in 1..N-1:
    if abs(SV[i] - SV[i-1]) > sv_jump_threshold:
        events.append(i)
```

阈值 `sv_jump_threshold` = max(0.5, 0.02 × span(SV))。窗口边界向后扩展至 `5 × T_prior` 或下一阶跃前 `merge_gap_s`，向前回溯 1 × dt。

**MV 阶跃检测**（mv_step）：

类似 SP 但作用于 MV，门限 = max(0.005 × span(MV), 1%)，需排除 SP 阶跃同时段（避免重复计入 SV 跟踪型窗口）。

**Rolling scan**：

滑动窗口长度 `L = clamp(int(60×T_prior/dt), 200, len/3)`，步长 `L/4`，对每个候选窗口算质量分。

#### 2.3.3 质量分（window_quality_score）

```
score = 0.30 × span_score      # MV 激励量级（min(mv_span / target_span, 1)）
      + 0.25 × corr_score      # |corr(MV, PV)| 经 r2-shrink 后的得分
      + 0.20 × monotonicity    # PV 单向运动占比（避免来回抖动）
      + 0.15 × stationarity    # 端点稳态分数（噪声 / span）
      + 0.10 × duration_score  # min(n_points / target_n, 1)
```

`window_usable_for_id = score >= 0.45 AND mv_span >= 0.05×range(MV) AND |corr| >= 0.30`。
权重与门限定义在 `policies/scoring_rules.py`。

#### 2.3.3 Policy 接入点

| Policy 字段 | 接入位置 | 行为 |
|---|---|---|
| `disabled_algorithm_families` | policy_composite | 关闭对应 family 计算 |
| `algorithm_plan[*].state` | history_rule_based | 标注 `policy_state`，driver 决定是否真跑 |
| `merge_gap_s` | policy_composite._merge_events | 合并间隔 |
| `expected_time_constant_range_s` | _apply_algorithm_plan | 影响 `mv_ramp` 的最小窗口长度 |

#### 2.3.5 异常路径

- 如果 `cleaned_df` 中 PV/MV 任一列方差 < 1e-9 → 返回空 `candidate_windows`，`meta.error="signal_zero_variance"`；
- 如果所有 family 都被禁用 → 直接返回空，runner 进入 `blocked_no_candidate_window` 分支（pipeline_status 标志）。

### 2.4 窗口选择（window_selection）

#### 2.4.0 输入 / 输出

- **输入**：`candidate_windows`、`window_policy`、`data_profile`、`selected_window_index?`、`use_llm_advisor`
- **输出**：`selection_meta`（mode / chosen_index / deterministic_index / window_candidate_decision / window_judgements / formal_identification_allowed）

#### 2.4.1 决策优先级

```
1. user_override：前端 selected_window_index 命中 → 直接采用
2. LLM advisor：use_llm_advisor=True 且 pool 大于 1
3. fallback_deterministic：LLM 返回非法
4. deterministic：LLM 关闭 / 池里只有一个窗口
```

#### 2.4.2 确定性默认（quality_score_selector）

- Skill：`select_window`（参数 `provider="quality_score_selector"`）
- 策略：在 `usable_windows` 集合内按 `window_quality_score` 取最大；空集时退回所有 `candidate_windows`；
- 输出：`chosen_index`、`score`、`chosen_window_summary`。

#### 2.4.3 LLM 顾问

`choose_window_via_llm`（`pipeline/llm_advisor.py`）：

- 输入：`data_profile`、`candidate_windows`（pool）、`loop_type`、`ontology_context`；
- 输出 JSON schema：
  ```json
  {
    "chosen_index": 0,
    "reasoning": "...",
    "ontology_evidence": [{"fact": "...", "source": "..."}],
    "window_judgements": [{"index": 0, "verdict": "preferred|acceptable|risk", "reason": "..."}]
  }
  ```
- 思考链通过 `llm_thinking{stage="window_selection"}` 单独事件发出。

#### 2.4.4 window_candidate_decision（数据分析菜单专用）

```python
{
  "selected_window_indices": [...],          # usable
  "rejected_window_indices": [...],          # 不 usable
  "fallback_window_indices": [...],          # 无 usable 时的诊断窗口
  "formal_identification_allowed": bool,     # 控制 Stage 2 闸门
  "diagnostic_identification_allowed": bool,
  "stop_reason": str | None,
  "primary_reason": str,
  "ontology_evidence": [...],
  "data_evidence": [{"fact","value","source"}],
  "window_judgements": [...],                # LLM 输出按 pool 索引映射回全局索引
  "recommended_identification_plan": {
    "mode": "formal" | "diagnostic_only",
    "model_types": ["FO","FOPDT","SOPDT"],
    "constraints": {expected_gain_sign, expected_time_constant_range_s, expected_dead_time_range_s}
  },
  "risk_flags": [...]
}
```

#### 2.4.5 异常路径

- 没有任何 `candidate_window` → 直接返回 `pipeline_status=blocked_no_candidate_window`；
- 没有 `usable_window`（`formal_identification_allowed=False`）→ 仍发完 window_selection done，但跳过 Stage 2；
- LLM 返回 `chosen_index` 越界 → 走 `fallback_deterministic`。

### 2.5 系统辨识（identification）+ 评审 + 精修

> 这是流水线最复杂的阶段。设计目标：让算法选出的"候选模型 ⊗ 候选窗口"组合中**最稳健**的胜出，而不是局部最优。

#### 2.5.0 输入 / 输出

- **输入**：`cleaned_df`、`pool`（usable + chosen 优先）、`dt`、`loop_type`、`data_profile`、`use_llm_advisor`
- **输出**（最终）：
  - `model: ProcessModel`
  - `confidence: ModelConfidence`
  - `attempts: list[dict]`（所有尝试明细）
  - `algorithm_comparison: list[dict]`（按 algorithm 聚合）
  - `model_review: {verdict, reason, concerns}`
  - `refinement_history: [...]`

#### 2.5.1 多轮"辨识 → 评审 → 精修"循环

```
MAX_REFINEMENT_ROUNDS = 2   # 即 1 次初辨识 + 至多 2 次精修
all_round_records = []
for round_idx in 0..MAX_REFINEMENT_ROUNDS:
    1. 准备本轮窗口集合（force_window_index 优先单窗）
    2. identify_model Skill → transfer_function_fit Provider
       → emit identification:done(round, attempts, algorithm_comparison)
    3. 若 LLM 关闭：直接 break（accept 默认）
    4. review_identification_via_llm
       → emit model_review:done(verdict in {accept, downgrade})
    5. 记录 round 快照
    6. verdict==accept 或 round==MAX → break
    7. ask_refinement_via_llm（失败回退 recommend_refinement_from_algorithm_comparison）
       → emit identification_refinement:done(force_window_index, force_model_types, hint_L)
       → 进入下一轮
end

# 跨轮收敛：
若整个循环没有 accept，按 (best fit_score, confidence) 选 round；
review_unreliable=True，会被传播到 evaluation 阶段
```

#### 2.5.2 transfer_function_fit Provider

- 入口：`Provider.identify(cleaned_df, candidate_windows, actual_dt, loop_type, ...)`
- 内部：
  1. `align_series` 对齐 PV/MV，去除起始稳态偏移；
  2. `detrend_if_needed` 删除线性趋势（基于残差白噪声检验）；
  3. `_estimate_dead_time`（cross-correlation，正向滞后）→ 初值 L₀；
  4. `_normalise` 归一化到 deviation variable（保留物理增益 K）；
  5. 按 `model_order_for_loop(loop_type)` 选择候选模型集合：
     - flow / pressure：`[FO, FOPDT, SOPDT, SOPDT_UNDER, IPDT]`
     - temperature：`[SOPDT, FOPDT, FO, IPDT]`
     - level：`[IFOPDT, IPDT, FOPDT, FO, SOPDT]`
  6. 对每个 (window, model) 调用 `scipy.optimize.least_squares`：
     - 参数边界由 `policies/constraints.py` 决定（K 同号、T ≥ min_reasonable_t、L ≥ 0、L ≤ 0.5×T_window）；
     - 初值由"单步阶跃法"先估计；
  7. 计算每个 attempt 的 `r2_score / normalized_rmse / fit_score`；
  8. AIC 比较：`AIC = N·ln(SSE/N) + 2k`，其中 k = `_N_PARAMS[model]`；
  9. 输出 `best_model`（按 `fit_score` 排序，AIC 仅在分差 < 1% 时启用）+ `attempts` + `fit_preview`。

#### 2.5.3 fit_score 公式

```
fit_score = 0.55 × r2_score
          + 0.25 × (1 - normalized_rmse)
          + 0.10 × window_quality_score
          + 0.10 × physical_plausibility
```

`physical_plausibility` 检查：
- `K` 与 `expected_gain_sign` 同号 → 1.0；不同号 → 0；未知 → 0.7；
- `T` 在 `expected_time_constant_range_s` 内 → 1.0，否则按距离衰减；
- `L < T` → 1.0，否则线性惩罚。

#### 2.5.4 评审顾问（review_identification_via_llm）

返回结构：

```json
{
  "verdict": "accept" | "downgrade",
  "reason": "...",
  "concerns": ["低 R²", "L 接近 0.5T", ...],
  "available": true,
  "reasoning_content": "<reasoner 思考链>"
}
```

`accept` 立即收敛；`downgrade` 触发精修。

#### 2.5.5 精修顾问（ask_refinement_via_llm）

返回结构：

```json
{
  "retry": true,
  "rationale": "...",
  "force_window_index": 2,
  "force_model_types": ["FOPDT","SOPDT"],
  "hint_L": 12.0,
  "source": "llm" | "deterministic_algorithm_policy"
}
```

确定性回退 `recommend_refinement_from_algorithm_comparison`：

1. 拿 `algorithm_comparison`（按 algorithm 聚合）；
2. 若当前最佳来自 `mv_step` 且 `r2 < 0.85`，建议尝试 `sp_step` 或 `mv_ramp` 来源的窗口；
3. 若 `L / T < 0.05`，建议 `hint_L = max(L, 0.5 × dt × ceil(prior_L / dt))`；
4. 若已退到 `rolling_scan` 且仍 downgrade，置 `retry=False` 让流水线进入 Phase 3 兜底。

#### 2.5.6 跨轮兜底（Phase 3）

```python
def _round_score(record):
    best_fit = max(a.fit_score for a in record.attempts if a.success)
    return (best_fit, record.confidence.confidence)
final = max(all_round_records, key=_round_score)
```

`review_unreliable=True` 在 evaluation 阶段拼到 `tuning_unreliable_reason`。

#### 2.5.7 异常路径

| 失败 | 行为 |
|---|---|
| Skill 调用失败 | 回退到 `algorithms.system_id.fit_best_model` 纯函数 |
| 整轮全部 attempts 失败 | `error_event(stage="identification", code="ID_ERROR")` |
| `confidence < 0.35` 且 round=0 且 LLM 关闭 | `error_event(code="LOW_CONFIDENCE")` |
| LLM 评审不可用 | `verdict=accept, fallback=true`，记录 `error_type / error_message` |

### 2.6 PID 整定（tuning）

#### 2.6.0 输入 / 输出

- **输入**：`model: ProcessModel`、`dt`、`loop_type`、`confidence`、`r2`、`nrmse`
- **输出**：
  - `best: {strategy, Kp, Ki, Kd, Ti, Td}`
  - `all_candidates: [...]`
  - `heuristic_strategy`、`heuristic_reason`
  - `tuning_unreliable / tuning_unreliable_reason`

#### 2.6.1 Provider 装配

- Skill：`generate_tuning_candidates`（默认 provider=`classic_family`）
- `classic_family` 内部委托：

| 子 Provider | 适用模型 | 适用回路 |
|---|---|---|
| `imc` | FOPDT / SOPDT | 流量、压力（小过冲优先） |
| `lambda_tuning` | FOPDT | 温度、液位（鲁棒优先） |
| `zn` | FOPDT / SOPDT | 通用，过冲较大；置信度低不建议 |
| `chr` | FOPDT | 跟踪 / 抑扰双口径 |

#### 2.6.2 主要公式（节选）

**IMC for FOPDT**（PI 形式）：

```
λ = max(0.5 × T, 1.5 × L)        # 默认保守 λ
Kp = T / (K × (λ + L))
Ti = T
Kd = 0
```

**Lambda for FOPDT**：

```
λ = lambda_factor × T            # lambda_factor 默认 1.0；保守 2.0 ~ 3.0
Kp = T / (K × (λ + L))
Ti = T
Kd = 0.5 × L                    # 仅在含微分动作时启用
```

**ZN（基于 K, T, L）**：

```
Kp = 1.2 × T / (K × L)
Ti = 2 × L
Td = 0.5 × L
```

#### 2.6.3 启发式推荐（heuristic_strategy）

```
if loop_type == "temperature":      推荐 Lambda（保守）
elif loop_type == "level":          推荐 Lambda（积分性）
elif confidence < 0.55 or nrmse > 0.25:  推荐 IMC，λ 调大 1.5×
elif loop_type == "flow":           推荐 IMC（小过冲）
else:                               推荐 IMC
```

`heuristic_reason` 是面向 LLM/用户的中文解释。

#### 2.6.4 Policy 接入

`policies/loop_priors.py`：

- `MIN_REASONABLE_T`：保护 ZN 在小 T 时 Kp 爆炸；
- `REALITY_T_RANGES`：评估阶段做现实性检查的范围。

#### 2.6.5 异常路径

- 模型不在 provider 支持集合（如 IPDT 给 ZN）→ 跳过该 provider，不抛错；
- 所有 provider 都失败 → 回退 `algorithms.pid_tuning.select_best_strategy`；
- 仍然失败 → 上抛 `ID_ERROR`。

---

### 2.7 评估（evaluation）

#### 2.7.0 输入 / 输出

- **输入**：`pid_params (Kp, Ki, Kd)`、`model_params`、`dt`、`loop_type`、`confidence`、`tuning_unreliable / reason`
- **输出**：
  - `passed: bool`
  - `performance_score: 0~100`
  - `final_rating: 0~5`
  - `overshoot_percent / settle_time_s / IAE / ITAE`
  - `time_series: {t, sp, pv, mv}`（前端绘图）
  - `reality_check: {issues: [...]}`

#### 2.7.1 Provider 装配

| Provider | 职责 |
|---|---|
| `closed_loop_sim` | 离散闭环仿真，给定阶跃 SP，返回时序 + 指标 |
| `closed_loop_response` | 解析法响应（用于交叉验证 sim） |
| `performance_scoring` | IAE / ITAE / 超调 / 调节时间归一化打分 |
| `reality_check` | 检查 `T` 是否落在 `REALITY_T_RANGES`，`L/T` 是否合理 |

Skill `evaluate_tuning` 默认链：`closed_loop_sim` → `performance_scoring` → `reality_check`。

#### 2.7.2 闭环仿真核心

- 对象：用 `model_params` 构造 G(s)；
- 控制器：并行 PID `u = Kp·e + Ki·∫e + Kd·de/dt`，含抗积分饱和（clamp）；
- 离散化：Tustin（双线性变换），步长 `dt`；
- 仿真长度：`T_sim = 8 × T + 4 × L`，最少 200 点。

#### 2.7.3 性能打分

```
score = 0.4 × inv(IAE_norm)
      + 0.3 × inv(overshoot_norm)
      + 0.2 × inv(settle_time_norm)
      + 0.1 × stability_margin
```

`passed = score >= 60 AND overshoot_percent <= 35 AND no_oscillation`。

#### 2.7.4 不可靠传播

`tuning_unreliable` 拼接：

```
combined_unreliable = tuning_result.tuning_unreliable OR review_unreliable
combined_reason = '；'.join([tuning_reason, review_reason])
```

`reality_check.issues` 中至少包含一项 → `passed=False`，`final_rating ≤ 2`。

#### 2.7.5 异常路径

- Skill 失败 → 回退 `algorithms.pid_evaluation.evaluate_pid_params`；
- 仿真发散（u 超过 1e6 或 NaN）→ `passed=False, performance_score=0, final_rating=0`，`reality_check.issues` 加 `simulation_diverged`。

## 3. Skill / Provider / Policy 装配关系

### 3.1 装配总览

```
┌─────────────────── Pipeline (runner.py) ────────────────────────┐
│ 阶段编排 + SSE 事件 + LLM 顾问触发                                │
└────────────────┬────────────────────────────────────────────────┘
                 │ registry.invoke(skill_name, args, ctx)
                 ▼
        ┌─────────────────┐
        │  Skill (LLM 视角) │  入参 Pydantic 校验、写入 LoopContext
        │  base.py + registry.py
        └────────┬────────┘
                 │ provider_registry.get(category, name)
                 ▼
        ┌─────────────────┐
        │ Provider (算法实现)│  纯算法，签名稳定，可多实现并存
        └────────┬────────┘
                 │ 调用
                 ▼
        ┌─────────────────┐
        │ algorithms / policies │  纯函数 + 规则常量
        └─────────────────┘
```

### 3.2 当前 Skill ↔ Provider 映射表

| Skill | 默认 Provider | category | 备注 |
|---|---|---|---|
| `load_dataset` | `dataset_loader` | common | 共享 Provider，不在 provider_registry，单独走 common |
| `summarize_data` | 内置 | data_understanding | 算法在 `_analyzers.py` |
| `detect_windows` | `policy_composite` | window_detection | 内部聚合 history_rule_based / event_window_builder |
| `select_window` | `quality_score_selector` | window_selection | 也可手动指定 |
| `estimate_dead_time` | `cross_correlation` | dead_time | 当前 transfer_function_fit 内自己用了类似逻辑，单独 Skill 用于诊断 |
| `identify_model` | `transfer_function_fit` | identification | 默认 |
| `generate_tuning_candidates` | `classic_family` | tuning | 内部委托 imc/lambda/zn/chr |
| `evaluate_tuning` | `closed_loop_sim` | evaluation | 链式调 performance_scoring + reality_check |
| `assess_loop_assessment` | 内置 | assessment | 静态评估，无外部 provider |
| `assess_loop_monitoring` | 内置 | monitoring | 监控页面用 |

### 3.3 注册机制

- **Skill**：`@register` 装饰器 + 子模块在 `core/skills/__init__.py` 被 import 时副作用注册；
- **Provider**：`@register_provider("category")` 装饰器，category 与 base 类的 `category` 属性必须一致；
- **运行时检索**：`registry.invoke("identify_model", args, ctx)` → Skill `run` 内 `provider_registry.get("identification", inputs.provider)`。

### 3.4 Policy 接入点

| Policy 文件 | 内容 | 被谁读 |
|---|---|---|
| `loop_priors.py` | `MODEL_ORDER / MIN_REASONABLE_T / REALITY_T_RANGES` | system_id.py、reality_check.py |
| `constraints.py` | 参数边界（K 同号、T 范围、L 范围、Kp/Ki/Kd 上下限） | least_squares 参数边界、tuning provider |
| `scoring_rules.py` | 窗口质量分权重、fit_score 权重、performance score 权重 | 窗口算法、identification、evaluation |
| `refinement.py` | 精修确定性回退规则 | identification_refinement_advisor |

### 3.5 新增一种 Provider 的接入步骤（开发指引）

1. 在 `providers/<category>/` 下新增 `my_provider.py`，继承对应 base class；
2. 实现签名要求的方法（如 `identify(...)`），保证纯函数；
3. 用 `@register_provider("identification")` 装饰类；
4. 在 `core/providers/<category>/__init__.py` 中 import 触发注册；
5. 测试：在 `tests/test_<provider>.py` 给至少 1 个黄金数据 + 1 个边界 case；
6. **不需要改 Skill**；调用方传 `provider="my_provider"` 即可切换。

## 4. 异常路径与降级策略汇总

### 4.1 阶段级错误码

| code | 触发位置 | 处理 |
|---|---|---|
| `DATA_ERROR` | data_analysis | 终止流水线，前端弹错 |
| `ID_ERROR` | identification | 终止流水线 |
| `LOW_CONFIDENCE` | identification round=0 且 LLM 关闭 | 终止 |
| `BLOCKED_NO_CANDIDATE_WINDOW` | window_selection（标志位，不是 error_event） | 早停 + result（pipeline_status） |
| `BLOCKED_NO_FORMAL_WINDOW` | identification 闸门 | 早停 + result（pipeline_status） |

### 4.2 LLM 顾问降级矩阵

| 决策点 | LLM 失败动作 | 影响 |
|---|---|---|
| ontology_policy | 用 base policy | family 计划仍可用 |
| window_selection | fallback_deterministic | 可能选到次优窗口 |
| model_review | accept + fallback=true | 不触发精修 |
| identification_refinement | recommend_refinement_from_algorithm_comparison（确定性） | 仍可重试 1 次 |

### 4.3 SSE 事件错误传播

- `error_event` 一旦发出，前端必须停止后续 stage 渲染；
- `result_event` 一定是流水线最后一条事件；
- `llm_thinking` 事件可独立出现，前端可折叠展示；
- 所有 stage_event 都应允许"乱序到达"的容错（前端按 stage 名称合并）。
