面向 PID 智能整定平台的下一阶段能力：实时监控最近 8 小时数据，结合本体工艺约束计算 Harris、CPK 等指标，完成诊断归因，并在满足安全条件时进入本体约束整定流程，最终输出可审计的整定建议。

1. 目标与边界

1.1 业务目标

  • 按装置、区域、回路实时扫描最近 8 小时历史数据，形成标准化评估快照。
  • 结合本体中的回路类型、变量含义、PV/MV/SP/DV、上下限、控制方向、阀门特性、工况规则、指标阈值，计算并解释 Harris、CPK、振荡、噪声、饱和、可辨识性等指标。
  • 诊断性能劣化原因：区分执行机构问题、测量问题、外扰/工况切换、PID 参数问题、模型不可辨识问题。
  • 当规则和风险门控判断需要整定时，自动创建整定任务，进入本体约束的选窗、辨识、评分、仿真和整定建议流程。
  • 最终输出可审计建议：是否需要整定、推荐 PID、仿真对比、风险说明、人工确认动作。

1.2 系统边界

能力是否自动边界说明
实时评估最近 8 小时数据自动低风险，可周期运行。
Harris / CPK / 诊断归因自动只读分析，不改变现场参数。
创建整定候选任务自动但受控需要满足数据质量、工况稳定、本体阈值和 workflow_guard。
生成推荐 PID 与仿真自动但受控可自动计算，但结果必须标注风险与依据。
下发 PID 到现场不自动默认不做。必须人工确认、权限校验和审批。
设计建议：产品上可以称为“评估智能体”和“整定智能体”，工程实现上使用一个统一的 RealtimeAssessmentSupervisor 编排标准 skill、规则和整定流水线，避免多智能体自由协商导致的安全边界不清。

2. 当前能力盘点

领域当前已有能力相关模块/接口缺口
历史数据 导入历史回路、获取 series/features/monitoring/assessment/windows。 /api/history/*，core/history/store.py 还不是面向实时周期评估的任务模型，缺少评估批次、状态、快照归档。
Harris/CPK 已有回路级 Harris、CPK 接口和前端性能页面使用。 /history/loops/{loop_id}/harris，/cpk 缺少统一评估快照表，缺少本体阈值驱动的自动判定。
本体/MCP 已有 MCP 服务配置、本体查询模板、整定 prior、本体规则。 /mcp-servers，/mcp-ontology-query-config，ontology_rules.py 需要把本体结果标准化落库，作为评估和整定统一输入。
诊断评估 已有 monitoring、assessment、risk alerts、diagnosis overview 页面基础。 assess_loop_monitoring，assess_loop_assessment 需要形成诊断归因模型：执行机构、测量、外扰、PID 参数、模型问题。
整定流程 已有本体策略、选窗、辨识、模型评审、整定候选、仿真场景、评估流水线。 run_tuning_pipeline，skill_orchestrator，workflow_guard 需要从评估结果自动创建整定任务，并把评估 evidence 传入整定流水线。
前端 风险预警、控制性能、诊断总览、模型可靠性、整定任务、对话设置。 RiskAlertsPanel，PerformanceScorePanel，TuningTask* 需要新增实时评估中心、自动整定任务队列、评估详情和调用轨迹。

3. 总体架构设计

3.1 逻辑架构

实时/历史数据源
  -> HistoryAdapter / HistorianAdapter
  -> RealtimeAssessmentScheduler
  -> RealtimeAssessmentSupervisor
       -> load_recent_loop_data
       -> resolve_ontology_context
       -> compute_harris / compute_cpk
       -> assess_loop_monitoring
       -> diagnose_root_cause
       -> decide_tuning_need
       -> workflow_guard
       -> create_tuning_task
       -> run_tuning_pipeline
       -> review_tuning_result
  -> 数据库快照与事件流
  -> 前端实时评估中心 / 风险预警 / 诊断总览 / 整定任务

3.2 后端分层

层职责建议模块
数据接入层按装置/回路/时间范围读取 PV/MV/SP/DV、清洗、重采样。core/history，新增 core/realtime/data_adapter.py
本体上下文层统一获取本体事实、约束、阈值、工况规则。core/ontology_rules.py，新增 core/realtime/ontology_context.py
指标计算层Harris、CPK、振荡、噪声、饱和、MV 行程、可辨识性。现有 history/store.py 能力沉淀为 assessment skills
诊断决策层执行机构/测量/扰动/工况/PID 参数/模型问题归因。新增 diagnose_loop_root_cause skill
整定触发层本体阈值 + 规则 + 风险门控，决定是否创建整定任务。新增 decide_tuning_need skill
整定执行层复用已有本体驱动整定流水线。core/pipeline/runner.py
审计与前端层保存评估批次、事件、调用轨迹、建议和确认状态。新增数据库表与 /api/realtime-assessment/*

3.3 核心设计原则

  • 指标计算尽量确定性，LLM 只做解释、冲突综合和建议文案。
  • 每个结论必须绑定 evidence，保存 skill 输入输出摘要。
  • 高风险动作由 workflow_guard 拦截，不允许 LLM 静默触发。
  • 实时评估和整定任务解耦：评估可以频繁跑，整定任务必须有状态、幂等和冷却时间。

4. 端到端流程设计

4.1 实时评估流程

1. 调度触发：每 5/15/30 分钟扫描启用的装置或回路，默认窗口最近 8 小时。
2. 数据读取：读取 PV/MV/SP/DV，校验采样间隔、缺失率、时间连续性。
3. 本体解析：获取回路类型、控制方向、变量角色、PV/MV 上下限、质量规格、典型时间常数、阀门约束、指标阈值。
4. 指标计算：计算 Harris、CPK、振荡、MV 饱和、MV 行程、噪声、扰动、可辨识性、整定准备度。
5. 诊断归因：基于本体规则和指标证据判断执行机构、测量、外扰、工况、PID 参数、模型问题。
6. 整定判定：满足阈值、数据质量和工况稳定时，生成 tuning_recommendation 或创建整定任务。
7. 结果落库：保存评估批次、快照、诊断、事件、建议和完整调用轨迹。

4.2 自动进入整定流程

assessment_snapshot.decision = "tuning_required"
  -> workflow_guard.check("create_tuning_task")
  -> create auto_tuning_task
  -> run_tuning_pipeline(
       loop_id,
       start_time,
       end_time,
       ontology_context,
       assessment_snapshot_id,
       trigger_mode="auto_assessment"
     )
  -> ontology_policy
  -> detect_windows + ontology filtering
  -> select_window
  -> identify_model
  -> review_identification
  -> generate_tuning_candidates
  -> build_simulation_scenarios
  -> evaluate_tuning
  -> final_recommendation

4.3 诊断归因规则示例

诊断类型证据组合建议动作
执行机构问题MV 饱和高、MV 频繁反向、死区证据高、PV 响应弱、阀门本体约束异常。检查阀门/执行机构，暂不直接整定。
测量问题PV 噪声高、跳变异常、传感器本体状态异常、Harris/CPK 受噪声主导。检查仪表、滤波和测点质量。
外扰/工况问题DV 变化明显、工况状态切换、SP/MV 无激励但 PV 波动。标记工况扰动，建议分工况评估。
PID 参数问题数据质量合格、执行机构正常、PV 振荡或长尾、Harris 劣化、CPK 低、模型可辨识。进入整定候选流程。
模型不可辨识MV 激励不足、窗口少、方向置信低、拟合 R2 低。建议寻找更好窗口或设计安全测试场景。

6.3 Supervisor 伪代码

async def run_assessment(loop_id, range_hours=8, trigger_mode="schedule"):
    ctx = LoopContext(loop_id=loop_id, time_range=last_hours(range_hours))
    trace = []

    for call in [
      "load_recent_loop_data",
      "resolve_ontology_context",
      "summarize_data",
      "assess_loop_monitoring",
      "compute_performance_indices",
      "assess_loop_assessment",
      "diagnose_loop_root_cause",
      "decide_tuning_need",
    ]:
        record = orchestrator.execute_one(call, ctx)
        trace.append(record)
        if not record.success:
            break

    snapshot = repository.save_assessment(ctx, trace)

    if ctx.data_profile["tuning_decision"]["decision"] == "tuning_required":
        guard = workflow_guard.check("create_auto_tuning_task", ctx, initiated_by="system")
        if guard.allowed:
            task = repository.create_tuning_task(snapshot)
            enqueue_tuning_pipeline(task)

    return snapshot

11. 安全与风控设计

11.1 风险等级

风险等级动作策略
low读取数据、获取本体、计算指标。可自动执行。
medium诊断归因、判断整定必要性、生成建议。可自动执行，但必须记录 evidence。
high创建整定任务、生成 PID 候选。系统可执行，但必须 workflow_guard 放行并保留人工确认。
critical下发参数或修改现场控制器。默认禁止，需权限、审批、回滚方案。

11.2 自动触发整定的硬性阻断条件

  • 当前工况为开停车、切换、联锁、异常工况。
  • 数据缺失率或时间不连续超过阈值。
  • MV 长时间饱和或执行机构疑似故障。
  • PV 噪声过高，Harris/CPK 结论不可置信。
  • 本体缺少关键变量角色或上下限，且无法从历史数据可靠推断。
  • 同一回路处于冷却期或已有未完成整定任务。

12. 实施计划

阶段 1：评估快照与只读闭环

  • 新增数据库表：assessment_job、assessment_snapshot、metric_result、diagnosis_result、assessment_event。
  • 新增 RealtimeAssessmentSupervisor，先支持手动触发单回路最近 8 小时评估。
  • 将现有 Harris/CPK/monitoring/assessment 结果统一写入评估快照。
  • 前端新增实时评估中心和评估详情抽屉。

阶段 2：本体阈值与诊断归因

  • 新增 ontology_context_snapshot，统一保存本体事实和阈值。
  • 新增 diagnose_loop_root_cause 和 decide_tuning_need skill。
  • 风险预警、诊断总览、模型可靠性页面接入评估快照。

阶段 3：自动创建整定任务

  • 新增 tuning_trigger、auto_tuning_task、tuning_recommendation 表。
  • 评估判定需要整定后，自动创建整定任务，但不自动下发参数。
  • 整定流水线接收 assessment_snapshot_id 和 ontology_snapshot_id，复用评估证据。

阶段 4：审批与闭环优化

  • 新增人工审批、驳回、复核和冷却策略。
  • 把最终结果回写评估快照，形成“评估 -> 整定 -> 再评估”的闭环。
  • 对话模式接入最新评估快照和调用轨迹，支持自然语言追问。

12.5 验收标准

验收项标准
实时评估单回路最近 8 小时评估可在 60 秒内完成，结果可复查。
本体融合每个评估快照能展示本体事实、阈值和来源。
诊断归因至少覆盖执行机构、测量、外扰、PID 参数、模型不可辨识五类。
整定触发能根据规则自动创建候选整定任务，且高风险动作不越权。
前端体验工程师能从风险列表一路追踪到指标、诊断、整定建议和调用轨迹。
