## 0. 文档说明

### 0.1 设计目标

1. 给出 V3.0 系统的**逻辑架构**与**模块边界**，让任何组员都能在一张图上知道自己在写哪一块；
2. 锁定**关键技术决策（ADR）**，把"不引入框架 / 算法纯函数 / Skill·Provider·Policy 三层 / 异步工程师确认 / 双流水线"等结构性决定记录在案；
3. 给出**横切关注点**（配置 / 日志 / SSE / 错误 / 并发）的统一规约，避免每个模块各写各的；
4. 给出**部署拓扑** / **性能容量假设** / **演进路线**，覆盖从 dev 到生产。

### 0.2 设计原则

- **确定性优先，LLM 是增量**：核心流程必须能脱离 LLM 跑通；
- **算法可替换**：算法层 = 纯函数；编排层 = Skill / Provider / Policy 注册可插拔；
- **数据契约稳定优先于代码复用**：跨层调用走 Pydantic 模型，不直接传 dict；
- **异步事件优先于阻塞调用**：所有跨阶段、跨人协同都用事件 + 重入；
- **可重放优先于热路径优化**：每次决策都落 jsonl，离线脚本可还原；
- **Karpathy 四原则**：先想后写 / 简洁优先 / 外科手术式改动 / 目标驱动闭环。

## 1. 架构目标与约束

### 1.1 业务目标

PRD § 1.3 已述。复述要点：

- 不当整定的回路绝不动；
- 每个数值都可追溯；
- 算法可替换、决策可重放；
- LLM 是可关闭的增量。

### 1.2 技术约束

| 约束 | 内容 |
|---|---|
| 后端语言 | Python 3.11+ |
| 后端框架 | FastAPI（保留 pid_v2 选型） |
| 计算栈 | NumPy / SciPy / Pandas |
| LLM 协议 | OpenAI 兼容（Qwen via DashScope / DeepSeek） |
| 前端 | React 18 + TypeScript + Ant Design Pro + Vite |
| 流式 | SSE（不引入 WebSocket） |
| 持久化 | 文件系统 `var/` + JSONL；不引入 RDBMS（V3.0） |
| 部署 | 单实例 + 进程级；可裸金属 / 容器二选一 |
| 外网依赖 | LLM API、可选；MCP server、可选；都允许失败降级 |

### 1.3 不变量

- 不引入 Agent 框架（AutoGen / LangChain / LangGraph）；
- 不直接对真实控制器下发参数；
- 不在缺关键本体数据时使用"猜测值"继续整定。

## 3. 分层与模块边界

### 3.1 分层职责

| 层 | 职责 | 不允许做的事 |
|---|---|---|
| 接入层（API） | HTTP/SSE 接口、入参校验、鉴权（V3.x）、错误码翻译 | 写算法、写持久化、调 LLM |
| 编排层（Pipeline） | 阶段顺序、SSE 事件、LLM 决策点、重入 | 实现具体算法、读 CSV、调 MCP |
| 能力层（Skill / Provider / Policy） | LLM 可调用的原子能力、可替换实现、规则常量 | 调 LLM（Provider 内禁；Skill 不主动调）、跨能力依赖 |
| 算法层（Algorithms） | 纯函数计算 | 任何 IO、状态、副作用 |
| Agent 层 | LLM 顾问、用户聊天 Agent | 直接计算（必须经 Skill） |
| 集成层（Services） | 外部系统调用、本地存储 | 业务规则 |
| 持久化（var/） | 文件系统 + JSONL | — |

## 4. 关键技术决策（ADR）

> 每条 ADR 含：决定 / 选项 / 选择 / 理由 / 后果。

### ADR-1：双流水线，不合并

**决定**：诊断与整定是两条独立的 Pipeline，各自启动、各自 SSE。

**选项**：
- A：单流水线，诊断作为前置阶段
- B：双流水线，参数通过 SessionContext 传递（**选**）
- C：诊断只产建议，整定流水线再决定是否引用

**理由**：
- 诊断的输出是"是否要整定"，与整定本身的计算无强耦合；
- 用户可能只想"看看诊断结果"，不进入整定；
- 重入挂起逻辑只发生在诊断；整定流水线保持线性 + 多轮精修；
- 单流水线把"挂起 + 重入"混进整定会让整定的状态空间爆炸。

**后果**：
- 诊断与整定通过 `session.parent_session_id` 双向引用；
- 整定流水线读 `diagnostic_verdict` 时若不存在不报错（兼容旧入口）。

### ADR-2：异步工程师确认 + 流水线重入

**决定**：confirmation 触发后 SSE 主动关闭流，工程师回应后调 `/sessions/{id}/resume` 触发重入。

**选项**：
- A：阻塞式（流水线挂在 await 上）
- B：异步事件 + 重入（**选**）

**理由**：
- 阻塞式需要长连接、资源不释放、超时困难；
- 异步事件可重连、可重启、可重放；
- 工程师可能离线 24h，挂起期间不应占用进程。

**后果**：
- 节点必须**幂等**（重入时用同样输入产同样结果）；
- 需要 SessionState 快照机制（见 § 5.5）；
- 前端需要"等待确认"卡片视图。

### ADR-3：Skill / Provider / Policy 保留三层

**决定**：保留 pid_v2 现有架构，扩展而非替换。

**理由**：
- 当前抽象与新需求方向一致：Skill 是 LLM 视角能力，Provider 是算法实现，Policy 是规则；
- 新增诊断节点 = 新增 Skill + Provider + 阈值，无架构变更；
- 这套抽象在 pid_v2 已经验证可用。

**后果**：
- Consultant 工具集需要从"直接调 algorithms"改为"透传到 SkillRegistry"，统一两套；
- 文档 6 全面更新。

### ADR-4：不引入 Agent 框架

**决定**：保留单 Consultant + 工具循环；4 个流水线 Advisor 单次结构化调用。

**理由**：
- 工具循环 ~80 行已能覆盖需求；
- 框架化会带来不可控的 prompt / 状态依赖；
- 失败回退在框架里更难写。

**后果**：
- 任何 LLM 调用都必须有确定性回退；
- 思考链作为独立 SSE 事件，不污染 stage_event。

### ADR-5：算法层强制纯函数

**决定**：`core/algorithms/` 不允许任何 IO、外部状态、全局可变量。

**理由**：
- 单元测试与回归极容易；
- 离线重放可在脚本里直接复刻；
- 多线程 / 多进程并发时不需要锁。

**后果**：
- IO（读 CSV、调 MCP）必须发生在 Skill / Provider / 集成层；
- 算法层签名稳定，签名变更必须 ADR。

### ADR-6：持久化用文件系统 + JSONL，不引入 DB

**决定**：V3.0 全部数据落到 `var/`（jsonl + json + sqlite for cache）。

**选项**：
- A：PostgreSQL / MySQL
- B：文件系统 + JSONL（**选**）
- C：MongoDB

**理由**：
- 单实例部署；
- 数据量小（每个会话 < 1MB）；
- jsonl 天然 append-only，便于审计；
- 离线重放只需读文件，不需要 db client。

**后果**：
- 跨进程一致性靠 SQLite WAL 或单进程访问；
- V3.x 多实例时换 PG（接口由 EventStore / SessionLog 抽象，迁移代价小）。

### ADR-7：SSE 不切 WebSocket

**决定**：流式协议保持 SSE。

**理由**：
- SSE 单向（服务端推），符合现状；
- 浏览器 EventSource 原生支持；
- 不需要双向时不引入 WS 复杂度。

**后果**：
- 工程师确认通过 HTTP POST 触发，不通过 SSE 上行；
- Consultant 流也是 SSE 单向，用户消息 POST。

### ADR-8：v3 长期分支策略

**决定**：在 `pid_v2` 仓库开 `v3` 长期分支演进，首次发布前不合并 main；发布时 main 切到 v3，旧 main 进 `legacy/v2`。

**理由**：
- 算法层资产共享 git history；
- 对比/回滚方便；
- 不开新仓库避免文档/CI 重建。

**后果**：
- M1-M4 期间 main 不接收破坏性变更；
- 紧急修复双轨：先 v2 main、再 cherry-pick 到 v3。

| MCP | `var/config/mcp_config.json` | MCP server 列表 |
| 阈值 | `var/config/thresholds.json` | 诊断阈值（可按 condition 覆盖） |
| 鉴权 | `var/config/credentials.json` | MCP token、未来 SSO |
| 功能开关 | `var/config/feature_flags.json` | LLM 总开关、诊断阶段开关 |

#### 热更新

- API `POST /api/config/{key}` 写文件 + 通知 in-memory store；
- 不重启进程；
- 写入审计落 `var/audit/config_changes.jsonl`。

### 5.2 日志

#### 字段规范

```
ts            ISO8601 with TZ
level         DEBUG / INFO / WARN / ERROR
trace_id      跨前端→后端→LLM 调用
session_id    pipeline session
pipeline      "diagnostic" | "tuning" | "consultant" | null
stage         (可选) 阶段名
event         "stage_running" | "skill_invoked" | ...
payload       结构化数据
```

#### 滚动 / 落点

```
backend/var/logs/
├── app.log                # INFO 及以上，按天滚动，30 天保留
├── debug.log              # DEBUG，仅 dev 环境，按 100MB 滚
└── audit.jsonl            # 用户敏感动作（写回、强制整定、配置变更）
```

#### 不记录

- LLM api_key
- MCP token
- 全量 PV/MV 时序（只记 first/last 100 点 + 统计）
- 用户 PII

### 5.3 错误处理

#### 错误码命名

`<DOMAIN>_<DETAIL>`，例如 `DATA_TOO_SHORT`、`ONTOLOGY_FIELD_MISSING`、`LLM_TIMEOUT`。

#### 错误事件 schema

```yaml
type: error
pipeline: diagnostic | tuning | null
stage: str | null
error_code: str
error_message: str          # 中文，给前端展示
error_details: object       # 调试用，可含 stack
trace_id: str
```

#### 异常处理三原则

1. **Skill 永不抛异常**（base 类已托底，转 SkillResult.success=false）；
2. **Provider 抛异常时立即被 Skill 转结构化结果**；
3. **API 层只把 error_event 翻译成 HTTP 状态码与 JSON body**，不做业务判断。

#### HTTP 状态码

| code | 用途 |
|---|---|
| 200 | 成功 / SSE 流（包含 error 事件不算 HTTP 失败） |
| 400 | 入参校验失败 |
| 403 | 权限/状态拒绝（如诊断 blocked 时强制整定） |
| 404 | 资源不存在 |
| 409 | 状态冲突（如 confirmation 已 expired） |
| 500 | 后端不可恢复异常 |
| 503 | 关键依赖不可达（仅在系统启动健康检查时使用） |

### 5.4 SSE 协议统一

#### 事件 envelope

```yaml
type: stage | llm_thinking | tool_call | alarm_raised |
      engineer_confirmation_required | result | error
pipeline: "diagnostic" | "tuning" | "consultant" | null
stage: str | null
status: "running" | "done" | null
event_id: str       # 单调递增，便于断线重连
ts: iso8601
trace_id: str
session_id: str
payload: object
```

#### 断线重连

- 客户端记录最后 `event_id`；
- 重连时带 `Last-Event-ID` header；
- 服务端从 jsonl 重放 N 条历史事件（最多 100 条）后续推送实时事件；
- 实现：`SessionLog` 提供 `iter_events_after(session_id, last_event_id)`。

### 5.5 会话状态与重入

#### Session 状态机

```
running → awaiting_confirmation → running → completed
running → completed
running → failed
running → cancelled
awaiting_confirmation → expired
```

#### 快照（snapshot）

每次进入 awaiting_confirmation 时落盘：

```
var/sessions/<session_id>/state.json
{
  "session_id": "...",
  "pipeline": "diagnostic",
  "status": "awaiting_confirmation",
  "current_node": "compute_cpk",
  "completed_nodes": ["identify_condition", "detect_open_loop", "estimate_snr"],
  "node_outputs": { "identify_condition": {...}, ... },
  "input_hash": "sha256:...",         # 输入数据摘要
  "confirmation_id": "...",
  "ctx_serialized": {...}             # LoopContext 关键字段
}
```

#### 重入逻辑

```
POST /api/sessions/{id}/resume
  ├── 加载 state.json
  ├── 校验 input_hash 与原始 csv 仍一致
  ├── 加载 confirmation.response
  ├── 重建 LoopContext（注入 confirmation.response）
  ├── 从 current_node 重跑（节点幂等保证结果一致）
  ├── confirmation.status = applied
  └── 继续 pipeline → 直至下一个 confirmation 或 completed
```

#### 节点幂等约束（细化）

- 节点不写 `var/`；持久化由 base 类统一在 invoke 之后做；
- 节点不调 LLM；LLM 决策点的回退路径必须无随机性，或随机种子来自 ctx；
- 节点签名 `(inputs, ctx) → SkillResult` 不能依赖时间戳（除非 ctx 提供）。

### 5.6 并发与线程模型

```
进程：1 个 (uvicorn)
事件循环：asyncio
计算密集任务：asyncio.to_thread（避免阻塞 loop）
LLM 调用：异步（OpenAI AsyncClient）
本体调用：异步（MCP client 已是 async）

线程池：CPU bound = ProcessPoolExecutor (V3.x 多回路并行时启用)
                  V3.0 单回路单会话，不需要进程池
```

#### 防并发问题

| 场景 | 措施 |
|---|---|
| 同一 session 重复 resume | 文件锁 + state.status 校验 |
| 同一 loop_id 多会话并发诊断 | 允许；每个会话独立 Loop ontology 缓存视图 |
| 配置热更新与读取 | 读不锁，写时写文件原子 rename + 通知 store reload |
| 告警写入与读取 | 一个写者（系统）、多个读者；index.jsonl append-only |

### 5.7 安全（V3.0 单租户基线）

| 维度 | 措施 |
|---|---|
| 鉴权 | V3.0 不强制；运行在内网；可选 Basic Auth |
| API 防滥用 | rate limit 100 req/min/IP |
| 写回操作 | 必须有 user 字段，落审计 |
| Prompt 注入 | 本体内容字段化注入 LLM，不直接拼接 |
| 密钥 | 不入日志、不返前端、不进 session 快照 |
| CORS | 仅允许同源 + 配置白名单 |

## 7. 性能与容量假设

### 7.1 单回路调用预算

| 阶段 | 无 LLM | 含 LLM |
|---|---|---|
| 诊断流水线（5 节点） | ≤ 5 s | ≤ 5 s（V3.0 不接 LLM） |
| 整定流水线（含 4 LLM） | ≤ 3 s | ≤ 30 s |
| Consultant 单轮 | — | P95 ≤ 12 s |
| 单 LLM Advisor | — | P95 ≤ 8 s |
| 单 MCP tool 调用 | ≤ 500 ms | — |
| SSE 首事件延迟 | ≤ 500 ms | — |

### 7.2 容量

| 指标 | V3.0 目标 |
|---|---|
| 同时在线会话数 | ≤ 20 |
| 每日诊断/整定调用 | ≤ 500 |
| var/ 增长率 | ≤ 50 MB/天（99% 是 jsonl） |
| 单数据集 CSV 大小 | ≤ 50 MB（10w 行） |

### 7.3 性能反模式（避免）

- 在算法层做 IO（违反 ADR-5）；
- 在 SSE 流里阻塞调用 LLM（必须 to_thread 或 async）；
- 把整段 CSV 序列化进 SSE event payload（只发统计 + 摘要）；
- 把 LoopContext 暴露给 LLM；
- 频繁本体调用（必须缓存）。
