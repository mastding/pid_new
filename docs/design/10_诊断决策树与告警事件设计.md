## 0. 文档说明

### 0.1 设计目标

1. 把 PRD 中的"诊断阶段"形式化为**确定性状态机 + 节点规约**，让任何实现都能 1:1 还原；
2. 定义两个一等公民事件（`AlarmEvent`、`EngineerConfirmation`），明确生命周期、API、SSE、持久化；
3. 把"诊断 → 整定"的衔接讲清楚，避免两条流水线耦合不当；
4. 所有节点遵循同一份模板：输入/输出 / 算法 / 阈值 / 决策动作 / 异常路径 / 测试用例；
5. 默认按 PRD 默认决策 D-1 ~ D-5 锁定 V3.0 范围。

### 0.2 设计原则

- **诊断只读不写**：诊断不修改回路状态、不下发参数、不写本体（V3.0）；
- **每个节点产出三类结果之一**：`continue`（继续）/ `alarm`（产告警）/ `confirmation`（产工程师确认事件，挂起）；
- **状态机外部可见**：诊断状态机的全部转移都通过 SSE 事件暴露，便于前端渲染与离线重放；
- **所有阈值有出处**：要么来自本体，要么来自配置中心，永远不在算法里硬编码；
- **可降级**：本体不可达时，诊断仍能跑完，但产 `risk_flag` 标注。

### 0.3 名词

| 名词 | 含义 |
|---|---|
| 诊断流水线（Diagnostic Pipeline） | V3.0 新增的、独立于整定流水线的状态机 |
| 节点（Node） | 决策树上的一个原子诊断步骤，对应一个 Skill |
| 边（Edge） | 节点之间的转移，由节点输出决定 |
| 裁决（Verdict） | 诊断流水线的最终结论，对应 `DiagnosticVerdict` 实体 |
| AlarmEvent | 告警事件，表示发现一个无需整定但需人工处置的问题 |
| EngineerConfirmation | 工程师确认事件，表示流水线缺关键工艺信息，挂起等待回应 |
| 重入（Resume） | 工程师确认回应后，流水线从挂起点恢复执行 |

### 2.1 Node-1：工况识别（identify_condition）

#### 2.1.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N1 |
| Skill 名 | `identify_condition` |
| 默认 Provider | `ontology_voting`（新增） |
| LLM 接入 | V3.0 否 |
| FR 关联 | FR-D1 |

#### 2.1.1 输入 / 输出

**输入**：

```yaml
loop_id: str
loop_type: str | null       # 可选
data_profile: object        # 来自 summarize_data
ontology_conditions: list[Condition]  # 来自本体；可能为空
```

**输出**（`SkillResult.data`）：

```yaml
condition:
  id: str                   # 命中工况 id；缺省 = "default"
  name: str
  confidence: float         # 0~1
  evidence: list[{fact, value, source}]
fallback_used: bool
risk_flags: list[str]
```

#### 2.1.2 算法逻辑

```
1. 如果 ontology_conditions 为空：
     return { id="default", confidence=0.5, fallback_used=true,
              risk_flags=["no_condition_in_ontology"] }

2. 对每个候选 condition c，计算匹配分：
     score(c) = mean over feature in c.match_features:
                indicator( data_profile[feature.key] in feature.range )
   其中 match_features 形如：
     [{ key: "pv_stats.mean", range: [120, 180] },
      { key: "mv_stats.mean", range: [40, 70] }]

3. 若 max(score) < 0.5：
     return { id="default", confidence=max(score), fallback_used=true,
              risk_flags=["condition_unknown"] }

4. 否则取 argmax 工况：
     return { id=c.id, name=c.name, confidence=score(c), evidence=[...] }
```

#### 2.1.3 阈值来源

| 阈值 | 默认 | 来源 |
|---|---|---|
| 命中阈值（0.5） | 0.5 | 配置 `diagnostic.condition.match_threshold` |
| 各工况 match_features | — | 本体 |

#### 2.1.4 决策动作

| 输出 | 动作 |
|---|---|
| `confidence ≥ 0.5` | `continue` 进 Node-2 |
| `confidence < 0.5` 但本体有定义 | `continue` 用兜底，risk_flag |
| 本体无定义 | `continue` 用兜底，risk_flag（V3.0 不挂起） |

> **设计说明**：V3.0 工况未识别不挂起流水线，因为后续节点的本体查询大多与具体工况无关。如果出现"必须知道工况才能查 USL/LSL"，由 Node-4 触发挂起。

#### 2.1.5 SSE 事件

```yaml
type: stage
pipeline: diagnostic
stage: identify_condition
status: running | done
payload:
  condition: {...}
  fallback_used: bool
  risk_flags: [...]
```

#### 2.1.6 异常路径

| 失败 | 行为 |
|---|---|
| 本体调用超时 | 走 fallback，risk_flag="ontology_unreachable" |
| Skill 内部异常 | `SkillResult(success=false)` → 整个诊断中止，发 `error_event` |

#### 2.1.7 测试用例

```
T-N1-1  本体定义 [低负荷, 高负荷]，PV 均值落在低负荷区间
        → condition.id="低负荷", confidence ≥ 0.7

T-N1-2  本体定义但所有工况都不命中
        → condition.id="default", confidence < 0.5, risk_flags=["condition_unknown"]

T-N1-3  本体未提供工况定义
        → condition.id="default", fallback_used=true, risk_flags=["no_condition_in_ontology"]

T-N1-4  本体调用超时
        → fallback，risk_flags=["ontology_unreachable"]
```

### 2.2 Node-2：开/闭环判断（detect_open_loop）

#### 2.2.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N2 |
| Skill 名 | `detect_open_loop` |
| 默认 Provider | `mv_change_window` |
| LLM 接入 | 否 |
| FR 关联 | FR-D2 |

#### 2.2.1 输入 / 输出

**输入**：

```yaml
cleaned_df: DataFrame      # 已清洗
dt: float
mv_range: float | null     # 来自本体；缺省时用 max(MV) - min(MV)
```

**输出**：

```yaml
is_open_loop: bool
max_delta_mv_pct: float    # 滚动窗口最大 ΔMV / 量程
duration_quiet_s: float    # 持续 ΔMV ≈ 0 的最长时长
evidence:
  window_start_idx: int
  window_end_idx: int
  mv_at_start: float
  mv_at_end: float
```

#### 2.2.2 算法逻辑

```
1. 用滚动窗口（长度 = 60s / dt，至少 12 点）扫描 MV：
     for each window:
        delta_mv = max(MV[w]) - min(MV[w])
        delta_mv_pct = delta_mv / mv_range
2. duration_quiet = 累计连续 ΔMV_pct < 0.001 (即 0.1%) 的总时长
3. 判定：
     is_open_loop = (duration_quiet ≥ 0.6 × total_duration)
                AND (max_delta_mv_pct < 0.005)  # 全局最大变化也很小
```

#### 2.2.3 阈值来源

| 阈值 | 默认 | 来源 |
|---|---|---|
| 滚动窗口长度（60s） | 60 | 配置 `diagnostic.open_loop.quiet_window_s` |
| ΔMV_pct 阈值（0.1%） | 0.001 | 配置 `diagnostic.open_loop.delta_threshold` |
| 安静持续比例（60%） | 0.6 | 配置 |
| 全局 ΔMV_pct 上限（0.5%） | 0.005 | 配置 |
| `mv_range` | 由本体 | 本体 `loop.mv_range`；fallback 用数据 |

#### 2.2.4 决策动作

| 输出 | 动作 |
|---|---|
| `is_open_loop = true` | 产生 `AlarmEvent("open_loop")`，verdict=blocked，**STOP** |
| `is_open_loop = false` | continue 进 Node-3 |

#### 2.2.5 SSE 事件

```yaml
type: stage
pipeline: diagnostic
stage: detect_open_loop
status: done
payload:
  is_open_loop: bool
  max_delta_mv_pct: float
  duration_quiet_s: float
  evidence: {...}

# 若开环，紧跟一个事件：
type: alarm_raised
payload:
  alarm_id: str
  alarm_type: "open_loop"
  severity: "high"
  recommended_action: "通知现场仪表工检查控制器是否在自动模式"
```

#### 2.2.6 异常路径

- 数据 < 60s（不够构造一个滚动窗口）→ `success=false, reasoning="数据时长不足"`，整个诊断 abort。

#### 2.2.7 测试用例

```
T-N2-1  正常变化的闭环回路 → is_open_loop=false
T-N2-2  MV 全程平直         → is_open_loop=true，触发告警
T-N2-3  MV 阶跃后长时间稳定 → is_open_loop=false（max_delta>0.5%）
T-N2-4  本体缺 mv_range，用数据 fallback → 仍能判定
```

### 2.3 Node-3：PV SNR 评估（estimate_snr）

#### 2.3.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N3 |
| Skill 名 | `estimate_snr` |
| 默认 Provider | `psd_ratio` |
| LLM 接入 | 否 |
| FR 关联 | FR-D3 |

#### 2.3.1 输入 / 输出

**输入**：

```yaml
cleaned_df: DataFrame
dt: float
loop_type: str
```

**输出**：

```yaml
snr_db: float
level: "high" | "medium" | "low"     # 信号质量等级
filter_recommendation:
  apply: bool
  type: "first_order" | "none"
  tau_s: float                        # 推荐时间常数
high_freq_power_pct: float
```

#### 2.3.2 算法逻辑

```
1. 对 PV 做 detrend（去线性趋势）
2. 算 PSD（Welch 方法），得到 power 分布
3. 定义"信号带"为低频段 [0, f_signal]，"噪声带"为高频段 [f_signal, f_nyquist]
   f_signal = 1 / (10 × T_prior)，T_prior 取 loop_type 默认（flow=5s, temp=200s 等）
4. signal_power = sum(power[0:f_signal])
   noise_power  = sum(power[f_signal:])
   snr_db = 10 * log10(signal_power / noise_power)
5. level / 滤波建议：
     snr_db ≥ 20  → high,    不加滤波
     10..20       → medium,  可选加 τ = 1×dt
     <10          → low,     建议 τ = 3×dt（按 loop_type 上限封顶）
```

#### 2.3.3 阈值来源

| 阈值 | 默认 | 来源 |
|---|---|---|
| signal/noise 频率界 | 1/(10×T_prior) | `policies/loop_priors.SNR_FREQ` |
| level 区间 | 20 / 10 | 配置 |
| τ 上限（按回路类型） | flow:5s, pressure:10s, temperature:30s, level:60s | `policies/loop_priors.SNR_TAU_CAP` |

#### 2.3.4 决策动作

| 输出 | 动作 |
|---|---|
| 任意 level | continue 进 Node-4 |
| `level=low` | continue + 注入 filter_recommendation 给后续整定（不阻断） |

#### 2.3.5 SSE 事件

```yaml
type: stage
pipeline: diagnostic
stage: estimate_snr
status: done
payload:
  snr_db: float
  level: str
  filter_recommendation: {...}
```

#### 2.3.6 异常路径

- PV 全程方差 < 1e-9 → snr_db=∞，level="high"（无意义但不报错）；
- 数据过短（< 200 点）→ 跳过 PSD，level="medium"，risk_flag="snr_insufficient_data"。

#### 2.3.7 测试用例

```
T-N3-1  纯正弦 + 弱噪声 → snr_db ≥ 25, level=high
T-N3-2  慢变化 + 强白噪声 → level=low, tau ≈ 3×dt
T-N3-3  数据 100 点 → 跳过 PSD，level=medium + risk_flag
```

### 2.4 Node-4：Cpk 计算（compute_cpk）

#### 2.4.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N4 |
| Skill 名 | `compute_cpk` |
| 默认 Provider | `cpk_basic` |
| LLM 接入 | 否 |
| FR 关联 | FR-D4 |

#### 2.4.1 输入 / 输出

**输入**：

```yaml
cleaned_df: DataFrame
loop_id: str
condition_id: str
ontology_pv_limits:        # 来自本体
  USL: float | null
  LSL: float | null
```

**输出**：

```yaml
cpk: float | null
cp: float | null
mu: float
sigma: float
within_limits_pct: float
recommend_tuning: bool      # cpk < 1.0 → true
threshold_used: float       # 默认 1.0
```

#### 2.4.2 算法逻辑

```
1. 若 USL is null OR LSL is null：
     return SkillResult(success=true, requires_confirmation="missing_pv_limits")
     → 触发 EngineerConfirmation，挂起诊断流水线

2. 取 PV 序列，剔除显著异常点（|x-μ| > 5σ）
3. 计算：
     μ = mean(PV)
     σ = std(PV)
     Cp  = (USL - LSL) / (6σ)
     Cpk = min((USL - μ) / (3σ), (μ - LSL) / (3σ))

4. recommend_tuning = (Cpk < 1.0)
```

#### 2.4.3 阈值来源

| 阈值 | 默认 | 来源 |
|---|---|---|
| Cpk 推荐整定门限 | 1.0 | 配置 `diagnostic.cpk.threshold`（按 condition 可覆盖） |
| 异常点 σ 倍数 | 5 | 配置 |

#### 2.4.4 决策动作

| 输出 | 动作 |
|---|---|
| 缺 USL/LSL | 产生 `EngineerConfirmation("missing_pv_limits")`，verdict=awaiting_confirmation，**挂起** |
| Cpk ≥ 1.0 | continue + 标记 recommend_tuning=false |
| Cpk < 1.0 | continue + recommend_tuning=true（不阻断，进 Node-5） |

#### 2.4.5 EngineerConfirmation 规约

```yaml
type: missing_pv_limits
prompt: "回路 {loop_id} 在工况 {condition_id} 下缺少 PV 上下限（USL/LSL），无法计算 Cpk。请补充："
fields:
  - { name: "USL", type: float, required: true, hint: "上规格界限，对应工艺允许的 PV 上限" }
  - { name: "LSL", type: float, required: true, hint: "下规格界限" }
  - { name: "writeback_to_ontology", type: bool, default: true,
      hint: "是否在工程师确认后写回本体（V3.0 仅写本地，V3.1 接 MCP）" }
expires_at: now() + 24h
```

#### 2.4.6 SSE 事件

```yaml
# 正常通路
type: stage
stage: compute_cpk
status: done
payload: { cpk, cp, mu, sigma, recommend_tuning, ... }

# 挂起通路
type: engineer_confirmation_required
payload:
  confirmation_id: str
  confirmation: { type, prompt, fields, expires_at }
# 紧跟 SSE 流主动关闭，会话状态 = "awaiting_confirmation"
```

#### 2.4.7 异常路径

- σ < 1e-6（PV 几乎不变）→ Cpk = +∞（无意义），recommend_tuning=false，risk_flag="pv_no_variation"；
- USL ≤ LSL → 数据校验错，产生 `AlarmEvent("invalid_ontology_data")`。

#### 2.4.8 测试用例

```
T-N4-1  USL=200, LSL=50, μ=180, σ=15 → Cpk≈0.44, recommend_tuning=true
T-N4-2  缺 USL              → 触发 EngineerConfirmation 挂起
T-N4-3  USL=100, LSL=200    → AlarmEvent("invalid_ontology_data")
T-N4-4  σ极小                → Cpk=∞, risk_flag
T-N4-5  工程师 24h 未回应    → confirmation expired，会话需重启
```

### 2.5 Node-5：Harris 指数闭环估计（compute_harris_closed_loop）

#### 2.5.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N5 |
| Skill 名 | `compute_harris_closed_loop` |
| 默认 Provider | `min_variance_ar` |
| LLM 接入 | 否 |
| FR 关联 | FR-D5 |

#### 2.5.1 输入 / 输出

**输入**：

```yaml
cleaned_df: DataFrame
dt: float
loop_type: str
condition_id: str
ar_order_max: int = 30      # AR 模型最大阶
deadtime_hint_s: float | null # Node-3 之前若有估值可传入
```

**输出**：

```yaml
harris_eta: float           # 0~1
confidence: float           # 0~1
deadtime_s: float           # 估计的 deadtime
ar_order_used: int
recommend_tuning: bool
threshold_used: float       # 默认 0.6
```

#### 2.5.2 算法逻辑

```
1. 对 PV 做 detrend
2. 估计 deadtime k：
     若 deadtime_hint 存在，用作初值
     用 SP-PV 误差 e = PV - SP（若无 SP 则用 ΔPV）
     找 e 自相关函数首个非显著点 → k
3. 拟合 AR(p) 模型，p 由 AIC 在 [2, ar_order_max] 内选：
     e(t) = a1·e(t-1) + a2·e(t-2) + ... + ap·e(t-p) + ε(t)
4. 计算最小方差预测残差：
     σ_mv² = var(ε[k:])           # k 步外的不可控部分
     σ_e²  = var(e)
5. Harris 指数：
     η = σ_mv² / σ_e²
   （注：行业有两种约定，此处 η 越大表示接近 MV 基准；
     部分定义 η = 1 - σ_mv²/σ_e²，越大越好。本系统采用前者，UI 时统一展示
     "η=0.78（值越大越好）"）
6. 置信度：
     confidence = clamp(  (data_length / (10 × deadtime_steps))
                        × (1 - residual_autocorr),
                        0, 1 )
7. recommend_tuning = (η < 0.6)
```

#### 2.5.3 阈值来源

| 阈值 | 默认 | 来源 |
|---|---|---|
| η 推荐整定门限 | 0.6 | 配置 `diagnostic.harris.threshold`（按 condition 可覆盖） |
| AR 最大阶 | 30 | 配置 |
| 置信度低门限 | 0.5 | 配置 |

#### 2.5.4 决策动作

| 输出 | 动作 |
|---|---|
| `confidence ≥ 0.5` | continue 进 Node-6 |
| `confidence < 0.5` | continue + risk_flag="harris_low_confidence"（V3.1 触发开环测试向导） |

> 即使 η ≥ 0.6（性能好）也仍 continue 到 Node-6，因为最终汇总要看所有节点。

#### 2.5.5 SSE 事件

```yaml
type: stage
stage: compute_harris_closed_loop
status: done
payload:
  harris_eta: float
  confidence: float
  deadtime_s: float
  recommend_tuning: bool
  risk_flags: [...]
```

#### 2.5.6 异常路径

- AR 拟合不收敛（数据噪声极高）→ confidence=0，η=null，risk_flag="harris_fit_failed"；
- 数据长度 < 10×deadtime → confidence ≈ 0，risk_flag="data_too_short"。

#### 2.5.7 测试用例

```
T-N5-1  良好 PI 控制（η≈0.85）        → recommend_tuning=false
T-N5-2  控制器 Kp 过小（η≈0.30）      → recommend_tuning=true
T-N5-3  数据 200 点，deadtime 50 点  → confidence<0.5, risk_flag
T-N5-4  AR 不收敛                      → η=null, risk_flag="harris_fit_failed"
```

### 2.6 Node-6：诊断结论汇总（summarize_diagnostic_verdict）

#### 2.6.0 元信息

| 字段 | 值 |
|---|---|
| 节点编号 | DIAG-N6 |
| Skill 名 | `summarize_diagnostic_verdict` |
| Provider | 内置（聚合逻辑） |
| LLM 接入 | 否 |
| FR 关联 | FR-D6 |

#### 2.6.1 输入 / 输出

**输入**：所有上游节点的 SkillResult.data。

**输出**：

```yaml
verdict:
  status: "healthy" | "needs_tuning" | "blocked" | "awaiting_confirmation"
  recommend_tuning: bool
  primary_reason: str
  blockers: list[AlarmEvent.id]
  confirmations: list[EngineerConfirmation.id]
  risk_flags: list[str]
  evidence:
    condition: {...}
    open_loop: {...}
    snr: {...}
    cpk: {...}
    harris: {...}
  recommended_tuning_hints:    # 给整定流水线的提示
    apply_filter: bool
    filter_tau_s: float
    skip_window_families: list[str]   # 例如 noise 高时跳过 rolling_scan
```

#### 2.6.2 算法逻辑（决策表）

```
1. 若 blockers 非空（任意节点产生 AlarmEvent + 标 blocking=true）:
     status = "blocked"
     recommend_tuning = false

2. 若 confirmations 非空且未 resolved:
     status = "awaiting_confirmation"
     recommend_tuning = false

3. 否则按以下规则计算 recommend_tuning：
     recommend_tuning = (cpk_recommend OR harris_recommend) AND NOT (cpk_excellent AND harris_excellent)

   细则：
     cpk_excellent       = cpk ≥ 1.33
     cpk_recommend       = cpk < 1.0
     harris_excellent    = η ≥ 0.8 且 confidence ≥ 0.6
     harris_recommend    = η < 0.6 且 confidence ≥ 0.5

4. 状态分配：
     recommend_tuning = true  → status = "needs_tuning"
     recommend_tuning = false → status = "healthy"

5. recommended_tuning_hints：
     apply_filter         = (snr.level == "low")
     filter_tau_s         = snr.filter_recommendation.tau_s
     skip_window_families = []   # V3.0 不动；V3.1 可由 SNR 影响
```

```yaml
  verdict: {...}

# 紧跟最终 result 事件
type: result
pipeline: diagnostic
payload:
  verdict: {...}
  alarms: [...]
  confirmations: [...]
  duration_ms: int
```

### 4.1 数据契约

```python
class EngineerConfirmation(BaseModel):
    id: str                          # uuid

    loop_id: str

    name: str

    required: bool = true
    hint: str = ""
    default: Any = None
    enum_values: list[str] | None = None
    min: float | None = None
    max: float | None = None
```

### 4.2 类型枚举（V3.0）

| type | 触发 | 字段 |
|---|---|---|
| `missing_pv_limits` | Node-4 缺 USL/LSL | USL, LSL, writeback_to_ontology |
| `missing_condition_def` | Node-1 严重不命中 | condition_id, override_thresholds |
| `missing_mv_range` | 多节点 | mv_min, mv_max |
| `confirm_window_choice` | 整定 window_selection 时 LLM 与确定性分歧严重 | chosen_index, comment |

### 4.3 状态机

```
   pending ───respond───► responded ───apply───► applied
      │                       │
      ├──cancel──► cancelled   │
      │                       │
      └────expire────► expired│
                              │
                     (apply 在重入流水线时由系统自动)
```

转移规则：

- `pending → responded`：通过 API `POST /api/confirmations/{id}/respond`，必填字段校验；
- `responded → applied`：流水线重入时，把 response 注入对应阶段，转 applied；如果重入失败，回到 `responded` 等待重试；
- `pending → expired`：超过 `expires_at`，定时任务（V3.0 简单 cron）扫一遍；
- `pending → cancelled`：用户主动放弃会话。

### 4.4 持久化

```
var/confirmations/<session_id>/<confirmation_id>.json
var/confirmations/index.jsonl
```

### 4.5 API

```
GET    /api/confirmations?status=&loop_id=
GET    /api/confirmations/{id}
POST   /api/confirmations/{id}/respond  body: {response: {...}, response_user}
POST   /api/confirmations/{id}/cancel   body: {reason}
POST   /api/sessions/{session_id}/resume   # 触发重入
```

### 4.6 重入机制

```
1. 流水线在节点 N 触发 confirmation：
   - 持久化 confirmation
   - 持久化 session "awaiting_confirmation" 快照（含已完成节点的 SkillResult）
   - SSE 推 engineer_confirmation_required，关闭流式连接
   - 返回 HTTP 200，body 含 confirmation_id

2. 工程师调 /respond：confirmation 转 responded

3. 调用方调 /sessions/{id}/resume：
   - 加载 session 快照
   - 把 confirmation.response 注入节点 N 的入参
   - 从节点 N 重新执行（节点须幂等）
   - 转 confirmation.status = applied
   - 重新建立 SSE 流式连接
```

### 4.7 SSE

```yaml
type: engineer_confirmation_required
pipeline: diagnostic | tuning
stage: <触发阶段>
payload:
  confirmation:
    id, type, prompt, fields, expires_at, evidence
```

### 4.8 节点幂等约束

任何可能触发 confirmation 的节点必须满足"用同样输入重跑结果一致"。具体要求：

- 节点不写持久化（写持久化的副作用集中在 Skill `invoke` 之外）；
- 节点接收的随机种子来自 `LoopContext.deterministic_seed`，由会话生成；
- 重入时 SkillRegistry 不重新触发 advisor / consultant。

## 5. 与整定流水线的衔接

### 5.1 数据传递

诊断流水线结束后产出 `DiagnosticVerdict`。前端在用户点"启动整定"时，把以下字段塞进整定流水线的 SessionContext：

```yaml
diagnostic_verdict:
  status: "needs_tuning"
  recommend_tuning: true
  recommended_tuning_hints:
    apply_filter: bool
    filter_tau_s: float
  evidence:
    condition_id: str
    cpk: float
    harris_eta: float
```

整定流水线在 `data_analysis` 阶段读取并：

1. 若 `apply_filter=true`：在 `signal_processing` 注入低通滤波；
2. 在 stage_event payload 里加 `diagnostic_evidence` 字段，便于前端联动展示；
3. 整定结束后，把整定结果回填到诊断会话日志（双向引用）。

### 5.2 不衔接的情况

| 场景 | 处理 |
|---|---|
| `verdict.status = healthy` | 整定按钮置灰；用户强制启动需勾选"忽略诊断结论 + 填写理由"，理由入审计 |
| `verdict.status = blocked` | 整定按钮置灰，提示告警先处置；强制启动按钮不显示 |
| `verdict.status = awaiting_confirmation` | 整定按钮灰色 + 进度提示；不允许强制启动 |
| 无诊断记录 | 旧版兼容：允许直接启动整定，但前端显示"建议先做诊断"提示 |

### 5.3 双向引用

```
DiagnosticVerdict.id  ←─── TuningResult.diagnostic_verdict_id
TuningResult.id       ───→ Session.tuning_result_id
                            ↓
                          DiagnosticVerdict.followup_tuning_session_ids: list[str]
```

## 7. 异常路径与降级矩阵

### 7.1 节点级失败

| 节点 | 失败原因 | 行为 |
|---|---|---|

| 失败 | 行为 |
|---|---|
| 数据库（var/）写失败 | error_event，abort |
| SSE 连接中断 | 客户端重连可恢复（事件流幂等 by event_id） |
| confirmation 持久化失败 | error_event，abort |

```

  → 计算输入 hash，与挂起时不一致 → confirmation.status = expired

```
