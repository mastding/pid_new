一、代码阅读结论

当前系统已经具备不少基础能力：历史数据导入、回路特征提取、监控快照、Harris/CPK、诊断规则、窗口选择、模型辨识、模型评审、仿真场景、候选 PID 生成、对话模式设置面板。 主要问题不是“没有能力”，而是这些能力分散在 API、store、runner、skill 与前端页面中，缺少统一的评估快照、统一的 skill trace、统一的自动整定任务入口。

当前文件/模块 已实现能力 主要问题 建议动作
backend/api/history_routes.py 历史回路、风险预警、Harris、CPK、本体规则、窗口、整定 SSE。 路由层承担了大量业务计算，风险预警、时间范围、资产筛选和整定准入耦合在一个文件里。 拆出 RiskAlertService、PerformanceMetricService、HistoryLoopQueryService，路由只负责参数校验和响应。
backend/core/history/store.py 文件型历史仓库、特征提取、监控、评估、Harris、CPK、窗口。 同时负责持久化、数据加载、指标计算、skill 调用和缓存，后续实时评估难以复用。 拆成 repositories/history_file_repository.py、services/loop_series_service.py、services/performance_metrics_service.py。
backend/core/ontology_rules.py 从 MCP 获取本体事实，结合数据质量、CPK、Harris、非整定故障输出规则。 本体事实解析、规则评估、指标计算调用混在一起，且结果还不是标准 skill trace。 拆成 resolve_ontology_context、evaluate_ontology_rules 两个标准 skill，并把规则包版本持久化到快照。
backend/core/pipeline/runner.py 完整整定流程：数据分析、本体策略、选窗、辨识、评审、整定、仿真评估。 仍是主流程大文件，但 skill 调用已统一收口到 SkillOrchestrator，SSE 已暴露默认 workflow_plan 事件；部分阶段仍直接调算法。 后续再把各阶段拆入 pipeline/stages，并将 LLM 插入/重试策略挂到同一个 orchestrator trace。
backend/core/pipeline/skill_orchestrator.py 已有默认模板、PlannedSkillCall、SkillExecutionRecord、guard 执行层。 已接入整定 runner 的 guard 执行入口与默认工作流展示；对话模式已展示启发式 skill 路由计划与元数据；实时评估快照已保存 assessment_template 工作流计划与执行 trace。 继续扩展为公共编排核心，支持 assessment_template、tuning_template、LLM 插入/重试/解释。
backend/core/workflow_guard.py 基于 preconditions 与 risk_level 限制 LLM 直接调用高风险 skill。 上下文字段判断有限，缺少 user_confirmed、trigger_mode、snapshot_id、权限策略。 升级为可配置 guard，区分 read_only、analysis、advisory、tuning_candidate、dispatch_task。
backend/core/skills/* 已有标准 skill 元数据：risk_level、preconditions、effects、stage、deterministic_gate。 监控/评估类 skill 缺少显式元数据；Harris/频域外部 skill 未完全纳入前端分组与自动流程。 补齐所有 skill 元数据，并新增实时评估、根因诊断、整定决策、任务下发 skill。
backend/api/assistant_routes.py 对话模式会默认读取 load_loop_context，把监控、特征、评估塞入上下文。 不是标准 tool/skill 调用，只是固定加载上下文；无法根据用户问题选择 Harris、CPK、本体、诊断或整定 skill。 改为 DialogueSupervisor：意图识别后调用只读 skill，再用 trace 回答。
frontend/src/features/dialogue/* 设置面板已集成 Skills、提示词、模型、MCP；对话可展示 thinking_step/tool_event。 显示的是模拟/摘要级事件，还不能展示统一的 skill 输入输出、耗时、风险门控与证据。 新增 SkillTraceTimeline 组件，展示 skill 名称、输入摘要、输出摘要、guard 决策、可展开详情。
frontend/src/features/loop-monitoring/* 风险预警、性能评分、诊断总览已有页面骨架和接口对接。 多处仍直接消费历史接口临时结构，缺少统一 AssessmentSnapshot 数据源。 切换到实时评估快照 API，保留原历史接口作为兼容 fallback。
frontend/src/features/model-reliability/ModelReliabilityPanel.tsx 能用已有字段计算模型可靠性展示。 可靠性分数偏前端拼装，没有绑定辨识模型版本、验证窗口、仿真场景与评审 skill 结论。 后端新增 model_review_snapshot，前端改为展示模型证据链，而不是只算静态分。
维护性债务： 部分后端源码中文注释和字符串在终端读取时出现编码异常。实施时不建议大面积格式化全部文件，但对本次触达的核心 skill、guard、assistant、ontology 相关文件，应在修改时顺手恢复为 UTF-8 可读文本。

二、目标流程拆解

用户目标可以拆成两个闭环：实时评估闭环和自动整定闭环。两个闭环共享本体上下文、历史数据窗口、skill trace、数据库快照，但风险级别不同。

1. 实时数据获取：按装置/回路定时获取最近 8 小时 PV、SP、MV、模式、报警、上下限、采样质量。
2. 本体上下文解析：从 MCP/本体获取回路类型、工况、量程、PV 规格限、MV 约束、阀门/执行机构信息、串级/前馈/MISO 关系。
3. 指标计算：统一计算 Harris、CPK、IAE/ISE、振荡、噪声、MV 饱和、执行器动作、可辨识性。
4. 诊断归因：结合本体规则判断问题更偏数据质量、测量噪声、执行机构、工况扰动、约束不足、PID 参数不当，避免所有问题都推给 PID。
5. 是否整定决策：只有当数据质量、工况代表性、约束裕度和本体策略通过后，才生成自动整定候选任务。
6. 整定任务执行：复用现有选窗、辨识、模型评审、候选参数、场景仿真、鲁棒性评估，但输入从评估快照继承。
7. 最终建议：输出“是否建议整定、推荐 PID、风险说明、仿真曲线、人工确认项、回滚建议”，默认不直接下发控制器。

三、后端改造清单

3.1 历史接口与实时评估服务拆分

步骤 新增/修改文件 动作 验收标准
1 backend/core/services/history_loop_query_service.py 承接 store.py 中按 loop_id、start_time、end_time 获取序列、特征、监控的查询逻辑。 /history/loops/{id}/features 与 /monitoring 响应不变，单测覆盖空回路、时间范围、缺列。
2 backend/core/services/performance_metric_service.py 集中封装 Harris、CPK、IAE/ISE、振荡等指标，Harris 仍调用 compute_harris_closed_loop 外部 skill。 性能页面和本体规则均从该 service 获取指标，不再在 store.py 内直接散落调用。
3 backend/core/services/risk_alert_service.py 迁移 history_routes.py 内联风险评分、风险分布、趋势和装置筛选逻辑。 /history/risk-alerts 只做参数解析；风险等级分布颜色、数量、趋势与旧接口一致或更完整。
4 backend/api/realtime_assessment_routes.py 新增实时评估 API，包括触发评估、查询快照、查询事件、自动整定候选。 前端可按装置/回路/最近 8 小时获取统一 AssessmentSnapshot。

3.2 从文件型 store 过渡到数据库快照

当前历史数据仍以 backend/var/history 文件为主。实施时不需要一次性推翻它，建议采用“文件仓库 + 数据库快照”的过渡方案：

  1. 原始 CSV/归一化历史继续由文件仓库管理，避免影响现有导入流程。
  2. 实时评估结果、指标结果、本体事实、诊断结果、自动整定任务写入数据库。
  3. 后续接入真实实时数据库或时序库时，只替换 LoopSeriesProvider，不影响 skill 和前端。

3.3 对话模式后端改造

当前行为 问题 目标行为
assistant_routes.py 默认构造 load_loop_context 事件，固定读取监控、特征、assessment。 用户问 Harris、CPK、本体、窗口、整定建议时，LLM 只能看固定上下文，不能按需调用标准 skill。 新增 DialogueSupervisor，先做意图分类，再选择只读/中低风险 skill 调用，最后基于 skill 输出回答。

五、编排与安全门控

5.1 不引入重型多智能体框架

当前更适合用单 Supervisor 编排标准 skill，而不是引入 AutoGen 这类多智能体框架。原因是 PID 整定流程有明确安全边界和确定性门控， 风险不在“智能体数量不够”，而在“每一步必须可审计、可回放、可阻断”。单 Supervisor + 标准 skill 已能满足实时监控、诊断、自动创建整定任务和对话解释。

5.2 编排模板

模板 默认步骤 允许 LLM 做什么 不允许 LLM 做什么
assessment_template load_recent_loop_data → resolve_loop_ontology_context → compute_performance_indices → diagnose_loop_root_cause → decide_tuning_need 解释、补充低风险查询、请求重算某指标、提示缺失字段。 直接创建整定任务或声称参数已可下发。
tuning_template load_dataset → build_ontology_policy → detect_windows → select_window → identify_model → review_identification → generate_tuning_candidates → build_simulation_scenarios → evaluate_tuning → review_auto_tuning_result 插入重试、解释失败原因、请求更保守仿真场景。 绕过数据质量、本体约束、模型评审和仿真评估。
dialogue_template classify_user_intent → choose_readonly_skills → answer_with_trace 按问题选择 Harris/CPK/诊断/窗口/整定建议 skill。 输出隐藏思维链原文；直接执行高风险整定任务。

5.3 workflow_guard 升级规则

  1. 将 risk_level 扩展为 low、medium、high、critical。
  2. 新增 initiated_by：system、llm、user、scheduler。
  3. 高风险 skill 需要 assessment_snapshot_id、ontology_context_id、guard_passed=true。
  4. 创建整定任务需要 trigger_mode 和 required_confirmations；自动监控只允许创建 pending_review 任务。
  5. 所有 guard 拒绝要写入 trace，前端可见，便于用户理解“为什么没有自动整定”。

阶段 目标 主要文件 完成标准

Phase 3 自动整定任务队列。 auto_tuning_routes.py、pipeline/runner.py、skill_orchestrator.py 评估快照可创建 pending 整定任务，runner 能继承 snapshot、本体策略和时间窗。
Phase 4 前端统一展示。 RiskAlertsPanel、DiagnosisPanels、ModelReliabilityPanel、dialogue/* 风险、诊断、模型可靠性、整定任务与对话模式共享同一评估快照和 skill trace。
Phase 5 定时调度与运行审计。 scheduler、repositories、settings 可按装置周期评估，自动产生风险和候选任务，所有动作可回放。
